# DRY Up extract-ruleset and update-ruleset Commands

**Date:** 2026-03-03
**Status:** Draft

---

## What We're Building

Extract ~140 lines of duplicate content shared between `.claude/commands/extract-ruleset.md`
and `.claude/commands/update-ruleset.md` into a single shared file. Fixes both edit overhead
(must update two files) and drift risk (one file falls behind the other).

Token cost is neutral: shared content is removed from each command file and loaded via a
read directive. Per-invocation line counts stay approximately the same.

---

## Why This Approach (Approach B — Extract Everything)

Approach A (identical sections only) would leave the 20-line multi-doc selection block and the
other pre-flight checks duplicated. These are the most likely to change as the pipeline evolves
(e.g., input-index.yaml format changes, new pre-flight guard added). Maximum DRY coverage is
worth the minor awkwardness of a split pre-flight.

---

## Key Decisions

### 1. Shared file location and name

`core/ruleset-shared.md` — lives alongside `civil-quickref.md` and `civil-authoring-constraints.md`.

### 2. What goes in the shared file

All content that is 100% identical or near-identical between the two commands:

| Section | Lines (approx) | Notes |
|---------|----------------|-------|
| Pre-flight checks 1, 3, 4, 5, 6 | ~65 lines | Split into Part A (checks 1) and Part B (checks 3-6) |
| Scoring Rubric | ~16 lines | Identical |
| CIVIL Reference (compact) | ~6 lines | Identical |
| Shared Procedures (Sub-A/B/C) | ~30 lines | Identical |
| Common Mistakes | ~10 lines | Identical |
| **Total** | **~127 lines** | |

### 3. Pre-flight split structure

The shared file defines pre-flight as two named parts:

```
### Pre-flight Part A
Check 1: Domain folder exists?

### Pre-flight Part B
Check 3: Input docs present?
Check 4: <filename> valid?
Check 5: Load ai-guidance.yaml
Check 6: Multiple input docs?
```

Each command file's `## Pre-flight` section becomes:
```
Read `core/ruleset-shared.md` first. Then run:

**Pre-flight Part A** (from shared file), then:

2. **[Command-specific check]**
   - ...

**Pre-flight Part B** (from shared file).
```

This preserves the numbered order (1, 2, 3–6) while keeping check 2 inline in each command.

### 4. What stays inline in each command

| Content | extract-ruleset.md | update-ruleset.md |
|---------|-------------------|-------------------|
| Header & description | ✅ | ✅ |
| Input section | ✅ | ✅ |
| Pre-flight check 2 | CIVIL file exists → redirect to /update-ruleset | CIVIL file missing → redirect to /extract-ruleset |
| Process steps (all) | ✅ (Steps 1–7c) | ✅ (Steps 0–9d) |
| Output table | ✅ | ✅ |

### 5. Read directive placement

Both commands start with:
```
Read `core/ruleset-shared.md` now. It contains shared pre-flight logic, the scoring rubric,
CIVIL reference, shared procedures (Sub-A/B/C), and common mistakes.
```

This appears before the `## Pre-flight` section in each command.

---

## Resolved Questions

- **Shared file name:** `core/ruleset-shared.md` — parallel to `civil-quickref.md`
- **Token cost:** Neutral — shared content removed from each command = same total tokens per run
- **Pre-flight structure:** Split into Part A / Part B to allow check 2 insertion
