# Machine-Readable CIVIL Schema via Pydantic Models

## What We're Building

`core/ruleset_schema.yaml` is currently 377 lines of pure `#` comments — `yaml.safe_load()` returns `None`. All valid values (types, precedences, rule kinds, etc.) are re-encoded as Python constants in `tools/validate_civil.py`, creating a dual source of truth that drifts silently.

We will replace this with Pydantic models (`tools/civil_schema.py`) that serve as the single authoritative schema for the CIVIL DSL. Pydantic generates JSON Schema automatically, wiring VS Code YAML autocompletion for free. The validator is rewritten to parse CIVIL YAML into Pydantic objects rather than checking dict keys manually.

## Why This Approach

Three approaches were considered:

- **JSON Schema file** — Hand-write and maintain `ruleset.schema.json` directly. Standard, but JSON is verbose and still requires separate Python validation logic.
- **Pydantic models (chosen)** — Define Python dataclass-style models; get JSON Schema and rich validation for free. Aligns with the existing Python toolchain. No hand-maintained JSON.
- **Structured YAML + generator** — Keep YAML as source of truth, add a `generate_schema.py` build step. More moving parts, non-standard format.

Pydantic was chosen because it eliminates the duplication (Python models replace both the YAML docs and the Python constants), generates editor tooling automatically, and keeps the project in one language.

## Key Decisions

- **Schema lives in Python**: `tools/civil_schema.py` contains Pydantic `BaseModel` classes for every CIVIL construct. This is the single source of truth.
- **JSON Schema is a generated artifact**: `make generate-schema` (or similar) runs `civil_schema.py` and writes `core/ruleset.schema.json`. Committed to the repo so VS Code works without a build step.
- **`validate_civil.py` rewrites to use Pydantic parsing**: `yaml.safe_load()` → `CivilModule.model_validate(data)`. Structural errors surface as Pydantic `ValidationError`. Custom `@field_validator` methods handle expression-language checks (`when:`, `expr:`) and mutual exclusions (`expr`/`conditional`).
- **Expression-language validation stays custom**: JSON Schema cannot validate CIVIL expression strings. `@field_validator` on the relevant fields carries this logic forward from the existing Python.
- **VS Code wired via workspace settings**: `.vscode/settings.json` maps `"*.civil.yaml"` to `core/ruleset.schema.json` using `redhat.vscode-yaml`. This covers all existing and future `.civil.yaml` files without per-file comments.
- **`transpile_to_opa.py` unchanged**: it reads CIVIL YAML by known key names and doesn't validate — no change needed.

## Resolved Questions

- **Fate of `core/ruleset_schema.yaml`**: Keep as a static narrative supplement. It documents the WHY behind the schema; `tools/civil_schema.py` is the authoritative source for valid values and structure. The two files can drift — that's acceptable since the YAML is explanatory, not prescriptive. The LLM prompt in `.claude/commands/extract-ruleset.md` continues to reference it unchanged.
- **Schema artifact location**: `core/ruleset.schema.json` — alongside `ruleset_schema.yaml` and `CIVIL_DSL_spec.md`.
- **`civil_schema.py` location**: `tools/civil_schema.py` — stays in `tools/` alongside the validator for easy import.
- **Pydantic version**: v2 (`model_validate`, `model_json_schema`) — no existing dependency to conflict with.
- **Test runner impact**: `run_tests.py` drives OPA tests from `_tests.yaml`, completely insulated from the validator — no change needed.
