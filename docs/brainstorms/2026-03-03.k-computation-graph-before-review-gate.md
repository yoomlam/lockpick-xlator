# Computation Graph Before Human Review Gate

**Date:** 2026-03-03
**Status:** Brainstorm complete

## What We're Building

Two related changes to `.claude/commands/extract-ruleset.md`:

1. **Graph preview before the human review gate**: Insert a new step (Step 6b in CREATE, Step 8b in UPDATE) between Validate and the Human Review Gate. This step runs the computation graph tool and surfaces the output inline during the gate.

2. **Shared Subroutines section**: Refactor the command to extract duplicated steps into a `## Shared Procedures` section at the bottom, referenced by both CREATE and UPDATE modes. This eliminates the current word-for-word duplication of Validate, Generate Computation Graph, and the "Extraction complete" footer.

## Why This Approach

The existing Step 7c runs graph generation *after* user approval, meaning the dependency visualization is unavailable during the review itself. Users reviewing complex programs (many computed fields, multi-step deduction chains) have no structural overview to catch missing dependencies or incorrect formulas. Generating the graph before the gate gives them exactly that context.

Step 7c is kept as a **refresh step** because the user may reject the initial extraction, triggering re-extraction of specific rules/fields. The CIVIL file changes during this process, so the pre-gate graph may be stale by the time approval is given.

The refactoring is motivated by the 763-line file currently duplicating Validate, Generate Graph, and the footer verbatim. Shared subroutines make future edits safer (single place to change) and the command easier to read.

## Key Decisions

### Step numbering
- New steps: **Step 6b** (CREATE) and **Step 8b** (UPDATE) — "Generate Computation Graph (Preview)"
- Existing **Step 7c** (CREATE) and **Step 9d** (UPDATE) remain as post-approval refresh steps
- Both 6b/8b and 7c/9d will reference the same shared subroutine (with different surrounding text)
- Minimizes renumbering churn

### Graph surfacing during the Human Review Gate
All three of the following appear **before the `Review summary:` header** (i.e., as an introduction to the gate, not interleaved with the rule buckets):

1. **Path notice** — `✓ Draft graph written: domains/<domain>/specs/<program>.graph.md`
2. **Dependency summary** — compact text block listing each computed field → its inputs and which rules use it (extracted from `.graph.yaml`)
3. **Mermaid diagram block** — full Mermaid `flowchart LR` rendered inline (VS Code preview renders it)

The ordering: path notice first, then summary, then Mermaid — summary is the fast-read, Mermaid is the full picture.

### Failure handling for Step 6b / Step 8b
Non-blocking: if `computation_graph.py` exits non-zero, print a warning and continue to the review gate without surfacing graph content. This matches the existing Step 7c/9d behaviour.

### Shared subroutines
Three blocks become shared subroutines referenced by both CREATE and UPDATE:

| Subroutine | Replaces |
|-----------|---------|
| **Sub-A: Validate** | CREATE Step 6 + UPDATE Step 8 |
| **Sub-B: Generate Computation Graph** | CREATE Step 7c + UPDATE Step 9d |
| **Sub-C: Extraction Complete Footer** | Both mode closing blocks |

Step 6b/8b logic (generate + surface) stays inline in each mode section since it involves the review gate — it is NOT a subroutine.

### Dependency Summary Format

Only the **computed fields** are shown (middle layer). Input and rule nodes appear implicitly as endpoints. Claude reads `.graph.yaml` — it does not re-parse the CIVIL file.

```
Computation graph (draft):
  adjusted_income    ← earned_income + unearned_income   → gross_income
  gross_income       ← adjusted + shelter_deduction      → [rule: FED-SNAP-DENY-001]
  net_income         ← gross_income - standard_deduction → [rule: FED-SNAP-DENY-002]
```

Each line: `<computed_field>  ← <depends_on list>  → <used_by list>`
Inputs listed without `[input]` prefix; rule sinks use `[rule: <id>]`.
