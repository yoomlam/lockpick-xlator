# extract-ruleset Decomposition

## What We're Building

Splitting `.claude/commands/extract-ruleset.md` (~680 lines) into three focused slash commands by applying the single responsibility principle. The goal is to reduce context window usage and clarify command boundaries.

The current command carries three distinct concerns in one file: (1) CIVIL extraction + validation + review, (2) test authoring, and (3) transpile + runtime testing. These will be separated.

## The Split

### What stays in `/extract-ruleset`

- Pre-flight checks (domain folder, input docs, OPA in PATH)
- **Makefile target scaffold** — moved from Step 9 to Step 1 (pre-flight), so it runs before any make commands are attempted
- Name Inventory (Step 3b) and naming manifest
- CIVIL file extraction + inline review scoring (Steps 1–6)
- Extraction manifest write/update
- Human Review Gate (Step 8 / Step 10)
- Handoff prompt: "Extraction complete. Next: run /create-tests, then /transpile-and-test. Continue? (y/n)"

Both CREATE and UPDATE modes are retained in `extract-ruleset`.

### New: `/create-tests`

Fully standalone, mode-aware command for test file maintenance. Detects domain and mode independently from repo state (presence of `specs/*.civil.yaml`).

- **CREATE mode** (no `*.civil.yaml` exists): write `specs/tests/<program>_tests.yaml` with 6+ cases covering happy path, gross deny, net deny, exemption, boundary, edge case; create `specs/tests/` silently if absent
- **UPDATE mode** (CIVIL file exists): identify stale test cases (inputs that no longer match updated thresholds/constants), rewrite stale cases, add new cases for new rules

### New: `/transpile-and-test`

Fully standalone, mode-agnostic command. Detects domain from cwd or optional `$DOMAIN` argument.

1. Detect domain
2. Run `make <domain>-transpile`
3. Run `make <domain>-test`
4. Relay make output verbatim; on failure ask user to diagnose — rule error vs test error vs transpiler bug

## Why This Approach

**Two commands vs. one `/finalize-ruleset`:** Granularity wins. Tests need re-drafting independently of transpilation (e.g., after manual CIVIL edits). Transpile+test needs re-running independently of test authoring (e.g., after fixing a Rego output bug). Combining them would force re-running steps unnecessarily.

**Makefile scaffold moved to pre-flight (not its own command):** It's a one-time setup check, not a user-facing action. Pre-flight is the right home — it's checked once at startup and never needs to be re-run manually.

**Ask at handoff (not auto-invoke):** The user may want to inspect the CIVIL file or review gate output before proceeding. An explicit y/n prompt respects that.

## Key Decisions

- **Handoff model**: `extract-ruleset` ends with "Continue? y/n" prompt after the review gate passes; user manually runs `/create-tests` then `/transpile-and-test`
- **Standalone commands**: `/create-tests` and `/transpile-and-test` detect domain and mode from repo state; usable at any time, not only immediately after `/extract-ruleset`
- **Makefile scaffold**: moves to pre-flight (Step 0) inside `extract-ruleset`, not its own command
- **No auto-invoke**: new commands are documented as "next steps" but not called automatically

## Expected Context Savings

Removing Steps 7, 9, 10 from CREATE and Steps 9, 11, 12 from UPDATE should reduce `extract-ruleset.md` by roughly 150–200 lines. The two new commands are each ~60–80 lines.

Net: three focused files instead of one large one, each smaller than the current single file.

## Resolved Questions

- **`specs/tests/` missing in CREATE mode**: create silently — no prompt or warning
- **`/transpile-and-test` output**: relay make output verbatim — no extra summary formatting
