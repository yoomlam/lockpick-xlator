# Slash Command Flow Audit

## What We're Building

A set of targeted fixes to the existing slash command suite — no new commands, no structural changes. The goal is to make the pipeline more reliable and consistent so a new user can move from `/new-domain` to `/transpile-and-test` without confusion or false starts.

---

## Issues Found

### 1. UX / Flow Gaps

#### 1a. `/new-domain` truncated next-steps list
**File:** `.claude/commands/new-domain.md`, Step 2

The next-steps output shows only 4 of 6 pipeline steps:
```
1. Add .md policy documents
2. Run /index-inputs
3. Run /refine-guidance
4. Run /extract-ruleset
```
Missing: `/create-tests` and `/transpile-and-test`. A new user following this guidance will think they're done when they're actually halfway through the pipeline.

**Fix:** Add steps 5 and 6 to the printed output.

#### 1b. Pre-flight ordering in `/extract-ruleset`
**File:** `.claude/commands/extract-ruleset.md`, Pre-flight steps 4–5

Step 4 is the expensive multi-file selection dialog (with index-aware prompts, file listing, `[1/2/.../a]` selection). Step 5 is the `ai-guidance.yaml` check — which **halts entirely** if the file is absent.

A user who hasn't yet run `/refine-guidance` will sit through the file selection dialog, then get a hard stop. Fail-fast ordering should put the `ai-guidance.yaml` check **before** the file selection dialog (swap steps 4 and 5).

#### 1c. Demo scaffold with no demo command
**Files:** `.claude/commands/new-domain.md`, `.claude/commands/transpile-and-test.md`

`/new-domain` creates `domains/<domain>/demo/` and the Makefile scaffold in `/transpile-and-test` adds a `<domain>-demo` target that runs `domains/<domain>/demo/start.sh`. But no command helps populate `start.sh`, and the docs don't explain how. The `demo/` folder is scaffolded but dead by default.

**Decision needed:** Should `/new-domain` emit a stub `start.sh`? Or should the Makefile target be dropped until a domain has a real demo?

### 2. Logic / Behavior Bugs

#### 2a. "Prompt then stop" anti-pattern
**Files:** `.claude/commands/index-inputs.md` (pre-flight check 1), `.claude/commands/refine-guidance.md` (pre-flight check 1)

Both commands say:
```
NO → List all directories... and prompt: "Which domain?" Then stop.
```

Prompting and then immediately stopping is contradictory — if you stop, the prompt has no effect. Compare to `/extract-ruleset` which says `"list... and prompt the user to choose"` (no stop), implying interactive handling.

This is a consistency bug. Either:
- All commands handle missing domain interactively (list options, await answer, continue), or
- All commands require the argument and stop with an error (no prompt, just: "Usage: /index-inputs <domain>")

The interactive approach is more ergonomic.

#### 2b. `--force` flag referenced but not implemented
**File:** `.claude/commands/extract-ruleset.md`, UPDATE Step 3

> "Run with --force to re-extract regardless."

No command syntax defines a `--force` flag. The input signature is `/extract-ruleset <domain> [<program>] [<filename>]`. This is documentation that describes behavior that doesn't exist.

**Fix:** Either add `--force` to the input signature and implement it, or remove the reference.

### 3. Cross-Command Inconsistencies

#### 3a. Missing domain argument: inconsistent handling
Three commands handle a missing `<domain>` arg differently:

| Command | Behavior |
|---------|----------|
| `/extract-ruleset` | List options, prompt interactively, continue |
| `/index-inputs` | List options, prompt... "then stop" |
| `/refine-guidance` | List options, prompt... "then stop" |

All three should behave the same way. Interactive prompting (like `/extract-ruleset`) is the better UX.

#### 3b. Inline `/index-inputs` execution: two different entry points
Both `/refine-guidance` (step 2) and `/extract-ruleset` (pre-flight step 4) offer to run `/index-inputs` inline if no `input-index.yaml` exists. The prompts are slightly different:

- `/refine-guidance`: `"Run /index-inputs <domain> now? [index / continue]"`
- `/extract-ruleset`: `"Run /index-inputs <domain> now? [y (recommended) / n — continue without index]"`

Two different option labels for the same semantic choice. Should be identical wording. Also: the `[index / continue]` option in `/refine-guidance` is non-obvious — typing "index" as a command response is unusual.

#### 3c. Makefile scaffolding: documented in wrong place
**File:** `.claude/commands/extract-ruleset.md`, Output table

The output table at the bottom of `/extract-ruleset` says:
```
| Makefile | Appended in pre-flight if no target existed | Not touched |
```

But `/extract-ruleset` has no Makefile-scaffolding steps anywhere in its process. That logic lives entirely in `/transpile-and-test` (pre-flight check 2). The output table in `/extract-ruleset` is stale/wrong.

**Fix:** Remove the Makefile row from `/extract-ruleset`'s output table, or add a note: "Handled by `/transpile-and-test`."

### 4. Documentation Accuracy

#### 4a. `/transpile-and-test` package derivation example is confusing
**File:** `.claude/commands/transpile-and-test.md`, pre-flight check 2

> "derive from its `module:` field (e.g., `"eligibility.snap_federal"` → `snap.eligibility`)"

The transform `eligibility.snap_federal` → `snap.eligibility` reverses the segments AND silently drops `_federal`. Neither the reversal logic nor the truncation is explained. The actual CIVIL schema uses `module: "eligibility.<program_name>"` and the Makefile var is `<domain>.<program>`. The example should be `"eligibility.snap"` → `snap.eligibility` (just a reversal, no truncation).

#### 4b. `/create-tests` UPDATE: stale-cases delete ordering
**File:** `.claude/commands/create-tests.md`, UPDATE Step 5

Step 4 writes the updated test file, Step 5 deletes `.stale-cases.yaml`. The ordering is correct but the instruction doesn't say "only delete if write succeeded." If Step 4 fails partway through, Step 5 would still clean up the hint file, losing the stale-case information. Minor robustness gap.

---

## Key Decisions

1. **Interactive vs. argument-required** — standardize missing-domain handling across all commands. Recommendation: interactive (list + prompt + continue), matching `/extract-ruleset`.

2. **Pre-flight ordering in `/extract-ruleset`** — move `ai-guidance.yaml` check before file selection. Simple reorder, no logic change.

3. **`--force` flag** — add it properly (with syntax documentation) or remove the reference. The flag is useful for UPDATE mode, so adding it is worth the effort.

4. **Demo scaffold** — either add a stub `start.sh` to `/new-domain`, or remove the `<domain>-demo` Makefile target from `/transpile-and-test` until a domain actually has a demo. The current "creates folder but no file" state is a dead end.

5. **Inline `/index-inputs` prompt wording** — unify `[index / continue]` vs `[y / n]` to the same phrasing.
