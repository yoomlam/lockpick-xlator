# Goal Files for `extract-ruleset` Prompt Engineering

**Date:** 2026-03-03
**Status:** Draft

---

## What We're Building

A two-layer prompt-engineering system for `/extract-ruleset`:

**Layer 1 — Domain-agnostic templates (`core/goals/*.yaml`):** Reusable goal files (e.g., `assess-eligibility.yaml`, `calculate-benefits.yaml`) defining variable taxonomy, role, scope, constraints, standards, guidance, and examples for a class of ruleset. Shared across all domains; not edited per-run.

**Layer 2 — Domain-specific instantiation (`domains/<domain>/specs/ai-guidance.yaml`):** A per-domain file derived from a selected goal template. Created on first run by copying the chosen goal file and prompting the user for customizations (known field names, domain-specific constraints, etc.). Reused on subsequent runs within the same domain.

**Flow:**
1. User runs `/extract-ruleset <domain>`
2. Command checks for `domains/<domain>/specs/ai-guidance.yaml`
   - **Exists:** use it directly as the pre-reading directive
   - **Missing:** present goal selection menu → user picks a goal → file is instantiated as `ai-guidance.yaml` with user customizations → proceed
3. The `ai-guidance.yaml` is injected into the extraction system prompt before the AI reads policy documents.

---

## Why This Approach

**Two-layer system:** shared goal templates + per-domain `ai-guidance.yaml`.

- The goal template stays domain-agnostic and reusable — it is never customized per program
- The `ai-guidance.yaml` layer lets users add domain-specific knowledge (known field names, program-specific constraints) without polluting the shared template
- This cleanly separates: *what class of ruleset is this?* (goal template) from *what do we already know about this domain?* (`ai-guidance.yaml`)

**Taxonomy of categories** (not explicit field names, not free-text) for the variable sections:

- Stays generic (works for SNAP, Alaska DOH, any other program)
- Is machine-parseable (structured YAML, not prose)
- Is comprehensive enough to capture all prompt engineering dimensions (role, scope, constraints, standards, guidance, examples)
- Avoids over-prescribing field names in the shared template (naming is handled by naming-manifest post-extraction)

---

## Key Decisions

1. **Scope:** Domain-agnostic templates, not domain-specific hints. The goal file says "look for income facts" not "look for `gross_income: money`".

2. **Timing:** Injected as a **pre-reading scoping directive** — before the AI reads policy documents, so it shapes attention, not just output format.

3. **Structure:** Combines three concerns in one file:
   - **Variable taxonomy** (`input_variables`, `output_variables`, `intermediate_variables`, `constants_and_tables`)
   - **Prompt engineering** (`role`, `scope`, `constraints`, `standards`, `guidance`)
   - **Examples** (`examples.ruleset_snippet`, `examples.test_case`)

4. **File placement:**
   - Templates: `core/goals/` (shared, not per-domain)
   - Instantiation: `domains/<domain>/specs/ai-guidance.yaml` (per-domain, git-tracked, user-editable)

5. **Discovery:** `/extract-ruleset` reads all `*.yaml` files under `core/goals/` and presents them by `display_name` when `ai-guidance.yaml` does not yet exist.

6. **Instantiation:** On first run, the selected goal template is copied to `domains/<domain>/specs/ai-guidance.yaml`. The user is then prompted to add domain-specific customizations (known field names, program-specific constraints, etc.) before extraction proceeds.

7. **Two-layer design rationale:** Goal templates are shared and never per-domain. `ai-guidance.yaml` is the mutable layer where users add domain knowledge. This keeps templates stable and reusable while allowing customization without duplication.

8. **Two initial goal files:**
   - `assess-eligibility.yaml` — for Boolean eligibility determinations (pass/fail)
   - `calculate-benefits.yaml` — for monetary benefit calculations (dollar amount output)

## Proposed YAML Schema

All goal files share this structure:

```yaml
goal_id: <identifier>                    # machine-readable key
display_name: "Human-Readable Name"     # shown in command selection menu
description: "..."                       # one sentence explaining the goal type

# === Prompt Engineering Blocks ===

role: "..."           # AI's purpose for this extraction task
scope: "..."          # focus statement: what to convert, what to produce

constraints:          # hard rules the AI must follow
  - "..."

standards:            # normalization rules (units, enums, formats)
  - "..."

guidance:             # edge cases, conflict resolution, domain-specific reminders
  - "..."

# === Variable Taxonomy ===

input_variables:
  categories:
    - category: <name>
      description: "..."
      examples: [field1, field2]

output_variables:
  primary:
    type: "bool"     # one of: bool, money, float, int — maps to CIVIL decisions[0].type
    description: "..."
  secondary_decisions:
    - name: "..."    # additional CIVIL decision fields (e.g., manual_review_required)
      type: "bool"
      description: "..."

intermediate_variables:
  categories:
    - category: <name>
      description: "..."
      examples: [field1, field2]

constants_and_tables:
  - name: <name>
    description: "..."

# === Examples ===

examples:
  ruleset_snippet: |
    # Short CIVIL YAML pseudocode example of desired rule logic
    # e.g., "If gross_income <= 130% of poverty_line, then passes_gross_test = true"
  test_case: |
    # Sample scenario + expected output
    # e.g., "Given household_size=3 and gross_income=$2000/mo, expect decision=true"
```

## `assess-eligibility.yaml` — Key Fields

| Section | Content |
|---|---|
| `role` | "Policy-to-rules analyst for eligibility determinations" |
| `scope` | "Convert policy text into explicit, testable eligibility rules" |
| `output_variables.primary.type` | `bool` |
| `input_variables` categories | household_composition, income, assets, citizenship, residency, program_history |
| `intermediate_variables` categories | income_deductions, gross_income_test, net_income_test, eligibility_flags |
| `constants_and_tables` | income_limits (by household size), deduction amounts |
| `constraints` | Do not interpret beyond the text; cite every rule; list gaps/unknowns; list assumptions made; do not invent verification requirements |
| `standards` | Monthly income amounts; dollar values; normalize citizenship status enums |
| `guidance` | Chained deductions (gross → earned deduction → net → shelter deduction → adjusted net); check for both gross and net income tests; when federal and state policies conflict, prefer the more restrictive rule |

## `calculate-benefits.yaml` — Key Fields

| Section | Content |
|---|---|
| `role` | "Policy-to-rules analyst for benefit amount calculations" |
| `scope` | "Convert policy text into explicit, testable benefit calculation rules that derive a dollar amount" |
| `output_variables.primary.type` | `money` |
| `input_variables` categories | household_composition, income, expenses, benefit_history |
| `intermediate_variables` categories | net_income, maximum_allotment, benefit_formula, proration_factor |
| `constants_and_tables` | maximum_allotment (by household size), minimum benefit threshold, thrifty_food_plan values |
| `constraints` | Do not interpret beyond the text; cite every rule; list gaps/unknowns; list assumptions made; ensure every monetary formula traces directly to a policy citation; do not round unless the policy explicitly specifies rounding |
| `standards` | Monthly amounts; dollar values; allotment tables keyed by household size; express percentages as decimals (0.3, not 30%) |
| `guidance` | Check for minimum benefit floors (do not compute amounts below the minimum); apply rounding rules only if stated; prorate for partial months only if stated; maximum allotment is a ceiling, not a formula — look for it in a table; benefit reduction = 30% of net income |

## Resolved Questions

- **Domain-agnostic vs domain-specific?** → Domain-agnostic templates. Field names are not prescribed; categories guide attention.
- **Pre-reading or post-reading injection?** → Pre-reading scoping directive.
- **File structure style?** → Taxonomy of categories + prompt engineering blocks (not field pattern lists, not free-text narrative).
- **What dimensions to include?** → role, scope, constraints, standards, guidance (merged strategies + reminders), examples, plus variable taxonomy (input/output/intermediate/constants).
- **`ai-guidance.yaml` vs `naming-manifest.yaml`:** No overlap. `ai-guidance.yaml` handles scoping/prompt engineering (what to look for). `naming-manifest.yaml` handles frozen field names (what was named). `ai-guidance.yaml` should reference `naming-manifest.yaml` rather than replicate field names. `naming-manifest.yaml` is already read by the command on UPDATE mode and should continue to be — independently of `ai-guidance.yaml`.
