# Computation Graph Tool

## What We're Building

A `computation-graph` command (Python tool + Makefile target + slash command) that reads a
CIVIL DSL YAML file and produces two files alongside it: `<ruleset>.graph.yaml` (structured
data) and `<ruleset>.graph.md` (Mermaid visualization). The graph documents every node in
the dataflow — input facts, computed intermediates, and rule sinks — with `depends_on` and
`used_by` links and YAML path locations.

Example: `domains/snap/specs/eligibility.civil.yaml`
→ `domains/snap/specs/eligibility.graph.yaml`
→ `domains/snap/specs/eligibility.graph.md`

## Why This Approach

The CIVIL DSL's `computed:` section is already in dependency order by contract, so topological
ordering is free. CIVIL expressions are valid Python syntax, so `ast.parse()` gives a robust
parse tree for dependency extraction — no fragile regex. Storing the graph as YAML keeps it
diffable, machine-readable for downstream tools, and consistent with the project's YAML-first
convention.

## Key Decisions

1. **Output location**: `domains/<domain>/specs/` — same folder as the CIVIL YAML. These are
   spec artifacts (document the design), not build artifacts, so they live alongside the source,
   not under `output/`.

2. **Node kinds**: Three kinds in the output YAML:
   - `input` — fact fields (`Household.earned_income`), constants (`EARNED_INCOME_DEDUCTION_RATE`),
     and table names (`standard_deductions`). Source nodes with no `depends_on`.
   - `computed` — intermediate values from the `computed:` section.
   - `rule` — deny/allow rules as sink nodes. Have `depends_on` but empty `used_by`.

3. **YAML output structure** (top-level):
   ```yaml
   domain: snap
   program: eligibility
   generated: 2026-03-03
   nodes:
     Household.earned_income:
       kind: input
       type: money
       location: facts.Household.earned_income
       depends_on: []
       used_by: [earned_income_deduction]
     net_income:
       kind: computed
       type: money
       description: "Net income after all deductions"
       location: computed.net_income
       depends_on: [income_after_prior_deductions, shelter_deduction]
       used_by: [passes_net_test]
     deny_net_income_exceeds_limit:
       kind: rule
       location: "rules[deny_net_income_exceeds_limit]"
       depends_on: [passes_net_test]
       used_by: []
   ```

4. **Location format**: YAML path notation — stable across line-number changes, human-readable.
   - Facts: `facts.Household.earned_income`
   - Computed: `computed.net_income`
   - Rules: `rules[deny_gross_income_exceeds_limit]`
   - Constants: `constants.EARNED_INCOME_DEDUCTION_RATE`
   - Tables: `tables.standard_deductions`

5. **Dependency extraction**: AST-based parsing via a new shared module `tools/civil_expr.py`.
   CIVIL expressions are valid Python syntax — use `ast.parse(expr, mode='eval')` and walk the
   AST to collect:
   - `ast.Attribute` nodes → fact references (`Household.earned_income`)
   - `ast.Name` nodes → computed field refs or constant/table name refs
   - `ast.Call` nodes where `func.id == 'table'` → table name from first string arg

   For `conditional:` fields, all three sub-expressions (`if`, `then`, `else`) are scanned
   independently and their references merged.

   `civil_expr.py` is extracted from `transpile_to_opa.py`'s existing expression-walking logic.
   Both `computation_graph.py` and `transpile_to_opa.py` import from it — this is a deliberate
   refactor of the transpiler, not just a copy. It makes expression semantics a single source
   of truth and is in scope for this feature.

6. **Node ordering**: Emit in dependency order — inputs first, then computed in CIVIL declaration
   order (guaranteed topological), then rules. Input nodes are collected in order of first
   reference across computed + rule expressions (no natural declaration order exists for inputs).

7. **Interfaces**:
   - `python tools/computation_graph.py <civil_yaml>` — Python tool, follows existing conventions
     (`fail()`, `load_civil()`, `✓` prefix on success, exit code 0/1)
   - `make <domain>-graph` — Makefile target added to each domain block
   - `/computation-graph <domain> [<program>]` — slash command at `.claude/commands/computation-graph.md`

8. **Visualization**: `<ruleset>.graph.md` contains a Mermaid `graph TD` diagram. Node shapes by kind:
   - Input nodes: rounded (`([...])`)
   - Computed nodes: rectangle (`[...]`)
   - Rule sink nodes: diamond (`{{...}}`)
   Edges flow from dependencies → dependents. Node labels include field name and type.

9. **Read-only**: Writes only `<ruleset>.graph.yaml` and `<ruleset>.graph.md`. No CIVIL, Rego,
   or test files are modified.

10. **`/extract-ruleset` integration**: As the final step of both CREATE and UPDATE modes (after
    the human review gate and naming manifest), `extract-ruleset` automatically runs
    `computation_graph.py` on the newly-written or updated CIVIL file. Every completed extraction
    produces an up-to-date graph with no separate invocation needed. Output:
    ```
    ✓ Computation graph updated: domains/<domain>/specs/<program>.graph.yaml
    ✓ Mermaid diagram updated:   domains/<domain>/specs/<program>.graph.md
    ```
    The `.graph.yaml` and `.graph.md` files are added to the Output table in `extract-ruleset.md`
    (auto-generated, not user-edited).
