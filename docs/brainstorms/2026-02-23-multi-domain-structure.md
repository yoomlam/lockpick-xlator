# Multi-Domain Project Structure Brainstorm

**Date:** 2026-02-23
**Status:** Draft

---

## What We're Building

A reorganized folder structure that supports multiple independent policy domains (e.g., SNAP, Medicaid, housing) within the xlator pipeline. Each domain is a self-contained unit with its own inputs, specs, outputs, and demo app. Shared tooling (transpiler, validator, test runner, CIVIL schema) lives at the project root.

---

## Why This Approach

The current layout is flat and SNAP-specific. The CIVIL DSL and schema already support multiple domains conceptually (dot-namespaced modules, jurisdiction layers), but the physical layout has not caught up. The **domain-first** structure (`domains/<name>/`) treats each domain as a first-class unit — matching how practitioners think about policy programs and making it trivial to add, hand off, or demo any single domain.

## Key Decisions

1. **Domain-first over pipeline-first** — Domains are the primary organizing concept. Each domain contains all four pipeline stages (input → specs → output → demo) rather than scattering a domain's files across four top-level directories.

2. **Separate demo per domain** — Each `domains/<name>/demo/` is a standalone FastAPI app with its own `start.sh`. Demos are not shared or routed through a single server.

3. **Shared tooling at root** — `tools/` and `specs/schema.yaml` stay at the project root. The transpiler and validator accept file paths as arguments and work across any domain.

4. **Independent programs (not jurisdictional variants)** — This structure is for completely separate policy domains. The CIVIL overlay system for jurisdictional variants (federal → state → city) is a separate concern handled within a domain's `specs/` directory.

5. **`output/` is generated** — Each domain's `output/` directory contains transpiled Rego. Like today, these are never hand-edited; regenerated by running `tools/transpile_to_opa.py`.

## Resolved Questions

- **Layout style:** Domain subfolders (not flat with prefixes)
- **Demo isolation:** Separate demo app per domain (not shared multi-route app)
- **Tooling scope:** Shared tools at root (not per-domain copies)
- **Domain type:** Independent programs (not jurisdictional variants)
- **Tool invocation:** Top-level `Makefile` with per-domain targets (e.g., `make snap`)
- **OPA scope:** One OPA instance per domain — each `domains/<name>/demo/start.sh` manages its own OPA process
- **`example_benefit.yaml`:** Delete — covered by `schema.yaml` and the SNAP domain
