# Brainstorm: `/create-demo` Slash Command

**Date:** 2026-03-03
**Status:** Draft

---

## What We're Building

A `/create-demo` slash command that generates a working FastAPI + browser demo for a domain's policy module. It reads the domain's CIVIL spec to produce tailored files in `domains/<domain>/output/demo-<module>/`.

---

## Context

The SNAP domain already has a hand-crafted demo at `domains/snap/output/demo-eligibility/` containing four files:
- `main.py` — FastAPI backend proxying to OPA REST API
- `static/index.html` — Browser form + JS rendering denial reasons and computed breakdown
- `start.sh` — Launches OPA + FastAPI on ports 8181 and 8000
- `requirements.txt` — fastapi, uvicorn, httpx, pydantic

This command automates creating that same structure for any new domain.

---

## Key Design Decisions

### 1. Generation Mode: Hybrid

Fixed file structure (4 files, always), with AI-populated domain-specific content drawn from the CIVIL spec:
- `inputs:` section → Pydantic model fields and HTML form inputs
- `program` + module name → OPA package path (e.g., `snap.eligibility`)
- `decision` output field → response structure
- Test cases from `specs/tests/<module>_tests.yaml` → 2–3 example scenarios; if absent, generate placeholder values and print a warning recommending `/create-tests` first

The slash command's `.md` file instructs Claude on the expected file structure; Claude fills in the domain-specific details. No template files to maintain.

### 2. Invocation: Auto-detect with Optional Args

Command signature: `/create-demo [domain] [module]`

- **With args** (e.g., `/create-demo snap eligibility`): skip scanning, proceed directly to that domain/module
- **Without args**: scan `domains/` for `.civil.yaml` files:
  - **One found**: proceed directly
  - **Multiple found**: present a numbered list, ask user to pick
  - **None found**: abort with helpful message pointing to `/extract-ruleset`

### 3. Existing Demo: Ask Before Overwrite

If `demo-<module>/` already exists, prompt:
> "Demo already exists at `domains/<domain>/output/demo-<module>/`. Regenerate and overwrite? [y/N]"

Abort on N, proceed on Y.

---

## Why This Approach

- **No template files**: Avoids a parallel set of files to keep in sync. The SNAP demo serves as the implicit canonical example referenced in the command prompt.
- **Auto-detect**: Consistent with how other commands (e.g., `/transpile-and-test`) resolve context — scan first, ask if ambiguous.
- **Hybrid generation**: More reliable than full free-form generation; the fixed file set guarantees expected outputs while AI handles domain-specific wiring.

---

## Command Execution Flow

1. **Pre-flight**: If args provided, use them directly; otherwise scan `domains/` for `.civil.yaml` files and resolve domain/module (ask if ambiguous)
2. **Read inputs**: Load CIVIL spec + test manifest (if present)
3. **Existence check**: If `demo-<module>/` exists, confirm overwrite
4. **Generate files**:
   - `requirements.txt` — static boilerplate (fastapi, uvicorn, httpx, pydantic)
   - `main.py` — FastAPI app with Pydantic models from CIVIL `inputs:`, OPA path from package name
   - `static/index.html` — Form fields from CIVIL `inputs:`, 2–3 examples from test cases
   - `start.sh` — Launcher with correct OPA package path
5. **Print next steps**: How to run (`bash .../start.sh`) and access the demo

---

## Out of Scope

- No `./x demo` CLI subcommand (slash command only for now)
- No live OPA connectivity check during generation
- No multi-module demos (one module per invocation)
- No update/patch mode (overwrite only)
