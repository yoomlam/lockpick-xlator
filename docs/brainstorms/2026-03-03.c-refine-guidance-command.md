# Brainstorm: `/refine-guidance` Command

**Date:** 2026-03-03
**Status:** Draft

---

## What We're Building

A new `/refine-guidance` command that owns the full lifecycle of `<domain>/specs/ai-guidance.yaml`. This file currently gets created inside the pre-flight section of `/extract-ruleset`. Extracting it into a dedicated command reduces `/extract-ruleset`'s length and complexity, and unlocks a richer guidance-authoring experience.

The key enhancement over the status quo: `/refine-guidance` is doc-aware. It reads the domain's `input-index.yaml` (if present), derives domain-specific signals from headings, summaries, and topic tags, and proposes initial guidance content before asking the user Q&A-style questions to refine it.

---

## Why This Approach (Approach C)

The doc-aware + Q&A model was chosen over simpler extraction because:

1. **Guidance quality scales with domain knowledge.** A bare goal template tells Claude to "look for income facts." A doc-aware file can say "look specifically for the earned income vs. unearned income split discussed in § 273.9, and the shelter deduction cap." The index already contains that information — using it is free.

2. **Q&A captures tacit knowledge.** Policy analysts know things the docs don't say explicitly (program history, common misapplications, state overrides). Structured questions surface that knowledge into durable YAML.

3. **`/extract-ruleset` becomes a pure consumer.** It reads `ai-guidance.yaml` but never writes it. Pre-flight becomes: does the file exist? If not, error with a clear message. This removes ~60-80 lines from `/extract-ruleset`.

---

## Key Decisions

### 1. `/refine-guidance` owns `ai-guidance.yaml` exclusively

`/extract-ruleset` will no longer create or modify this file. Pre-flight in `/extract-ruleset` becomes:
```
If domains/<domain>/specs/ai-guidance.yaml does not exist:
  STOP. Tell user: "Run /refine-guidance <domain> first."
```

### 2. Two modes: CREATE and UPDATE

- **CREATE:** Domain has no `ai-guidance.yaml`. Run full flow: goal selection → doc analysis → Q&A → write file.
- **UPDATE:** File already exists. Run doc analysis (re-check index for new material) → Q&A to refine specific sections → overwrite file.

Mode is detected by presence/absence of the file (consistent with other commands).

### 3. Doc analysis reads `input-index.yaml` (soft prerequisite)

If `specs/input-index.yaml` exists, `/refine-guidance` extracts:
- Topic tags across all sections (clusters signal what domain areas are prominent)
- Section headings (reveals statutory structure)
- File summaries (reveals program scope)

From these signals, it proposes concrete additions to each guidance section before Q&A starts.

**If `input-index.yaml` is absent:** ask the user, similar to how `/extract-ruleset` pre-flight handles a missing Makefile target:
> "No input index found for this domain. Run `/index-inputs <domain>` first to enable doc-aware suggestions, or continue without doc analysis?"

- If user says "continue" → proceed with goal-template defaults only (no doc-derived proposals).
- If user says "index first" → execute the `/index-inputs` flow inline for this domain, then resume doc analysis and continue.

### 4. Q&A structure covers four guidance dimensions

Each round of Q&A targets one section of the YAML:

| Section | Key question |
|---|---|
| `constraints` | "What should I *not* infer or assume in this domain?" |
| `standards` | "Are there normalization rules specific to this program? (units, categories, naming)" |
| `guidance` | "What non-obvious rule patterns should I look for?" |
| `edge_cases` (new section) | "What special populations or situations does this program treat differently?" |

Q&A is interactive — Claude asks one question per section, shows its doc-derived proposal, and the user confirms/amends/adds. All four sections are always asked in sequence; the user can answer "nothing to add" to skip any section.

### 5. New `edge_cases:` section in `ai-guidance.yaml`

The goal templates have `guidance:` (general hints). We add a top-level `edge_cases:` list specifically for program-specific populations or situations that require special handling. This is the primary output of the Q&A phase and has no equivalent in the current goal-template schema.

### 6. Goal selection stays in CREATE mode

The goal-file selection menu (currently in `/extract-ruleset` pre-flight) moves to `/refine-guidance` CREATE mode. Goal files (`core/goals/*.yaml`) remain the starting template; doc analysis and Q&A layer on top.

## Command Flow

### CREATE Mode

```
Pre-flight:
  1. Domain folder exists?
  2. No ai-guidance.yaml present (else → UPDATE mode)

Step 1: Goal selection
  Present menu of core/goals/*.yaml. User selects one.
  Copy selected goal as base.

Step 2: Doc analysis (if input-index.yaml exists)
  - Read specs/input-index.yaml
  - Extract topic tags, headings, file summaries
  - Generate proposed additions for each guidance section

Step 3: Q&A refinement
  For each section (constraints, standards, guidance, edge_cases):
    - Show doc-derived proposal (if any)
    - Ask the section's key question
    - Incorporate user answer into the section

Step 4: Write ai-guidance.yaml
  - Include source_template: <goal_id>
  - Include generated_at: <date>
  - Write full refined content

Output:
  domains/<domain>/specs/ai-guidance.yaml  [CREATED]
```

### UPDATE Mode

```
Pre-flight:
  1. Domain folder exists?
  2. ai-guidance.yaml exists (else → CREATE mode)

Step 1: Load existing file
  Show current contents summary to user.

Step 2: Re-run doc analysis (if input-index.yaml exists)
  Check for new topic areas not covered in existing guidance.
  Flag any apparent gaps.

Step 3: Q&A refinement (targeted)
  - Show user any new gap areas found by doc analysis
  - Ask all four section questions in sequence (same as CREATE mode)
  - User can answer "no changes" to skip any section
  - Only sections where the user provides input are rewritten; others preserved verbatim

Step 4: Write updated ai-guidance.yaml
  - Preserve source_template
  - Update generated_at

Output:
  domains/<domain>/specs/ai-guidance.yaml  [UPDATED]
```

## Impact on `/extract-ruleset`

**Removed from pre-flight:**
- Goal selection menu
- `ai-guidance.yaml` creation flow
- User customization prompt

**Remaining in pre-flight:**
- Domain folder check
- Input docs check
- Makefile target scaffold
- Load `ai-guidance.yaml` (read-only; error if absent)

**Net reduction:** ~60-80 lines from the pre-flight section. The command becomes strictly a CIVIL extractor — it never manages its own configuration.

---

## `ai-guidance.yaml` Schema Change

Add `edge_cases:` as a top-level list (parallel to `guidance:`):

```yaml
# Existing sections unchanged
goal_id: assess-eligibility
source_template: assess-eligibility
generated_at: "2026-03-03"
role: "..."
scope: "..."
constraints: [...]
standards: [...]
guidance: [...]

# New section
edge_cases:
  - "Households with elderly or disabled members get expanded categorical deductions"
  - "Mixed-status households include citizens/LPRs only in household size for limit lookup"
```

The `edge_cases:` list is injected into the extraction prompt as a distinct block: "Pay particular attention to these edge cases."
