# feat: Extract `/refine-guidance` Command from `/extract-ruleset`

## Overview

Refactor the `ai-guidance.yaml` lifecycle out of `/extract-ruleset` into a dedicated `/refine-guidance` command. The new command adds doc-aware AI suggestions and structured Q&A on top of the existing goal-selection flow. `/extract-ruleset` becomes a pure consumer: it reads the file or stops with a clear error.

## Motivation

Pre-flight section 5 in `/extract-ruleset` (lines 83–105) mixes guidance authoring with extraction setup. It's simple today, but it's the wrong place for interactive Q&A. Extracting it:

- Reduces `/extract-ruleset` by ~25 lines in pre-flight alone
- Gives guidance authoring its own command with room to grow
- Enables doc-aware proposals without making `/extract-ruleset` longer

## Files Changed

| File | Action |
|------|--------|
| `.claude/commands/refine-guidance.md` | **CREATE** — new command |
| `.claude/commands/extract-ruleset.md` | **MODIFY** — replace pre-flight section 5 |

No other files need changing. Goal templates (`core/goals/*.yaml`) are unchanged.

## Implementation

### 1. Create `.claude/commands/refine-guidance.md`

Follow the structural pattern established by `/index-inputs` and `/create-tests`:

```
# Header + description
## Input
## Pre-flight
## Mode Detection
## Process — CREATE Mode
## Process — UPDATE Mode
## Output
## Common Mistakes to Avoid
```

#### Input

```
/refine-guidance <domain>
```

#### Pre-flight

1. **Domain provided?** — If not, list `domains/*/` and prompt.
2. **Domain folder exists?** — If not, print error and stop. (Do not scaffold; that's `/new-domain`'s job.)
3. **Detect mode** — `domains/<domain>/specs/ai-guidance.yaml` present → UPDATE; absent → CREATE.

#### Process — CREATE Mode

**Step 1: Goal selection**

Scan `core/goals/*.yaml`:
- No files found → print warning, continue without a base template
- Exactly one found → print: `One goal template found: <display_name>. Using it.` Confirm with user.
- Multiple found → numbered menu of `display_name` values; user selects one

Print the selected template content for review.

**Step 2: Input index check**

Check for `domains/<domain>/specs/input-index.yaml`:

- **Exists** → read it; proceed to Step 3 (doc analysis)
- **Absent** → ask:
  ```
  No input index found for this domain. An index enables doc-aware guidance suggestions.
  Run /index-inputs <domain> first, or continue without doc analysis? [index / continue]:
  ```
  - `index` → execute the `/index-inputs` flow inline for this domain, then proceed to Step 3
  - `continue` → skip to Step 4 (Q&A with goal-template defaults only)

**Step 3: Doc analysis** *(only if input-index.yaml available)*

Read `specs/input-index.yaml` and extract signals:
- **Topic tags** across all sections → cluster to find prominent domain areas
- **Section headings** → reveals statutory structure (e.g., income tests, deduction chains)
- **File summaries** → reveals program scope and terminology

For each of the four guidance sections (`constraints`, `standards`, `guidance`, `edge_cases`), generate a list of proposed additions grounded in the index signals. Hold these proposals for Step 4.

**Step 4: Q&A refinement**

Ask one question per section, in order. For each:
1. Show doc-derived proposals (if any from Step 3), formatted as a bullet list
2. Ask the section's key question (see table below)
3. Incorporate the user's answer: if they confirm, keep proposals; if they add text, append it; if they replace, use their text

| Section | Key question |
|---|---|
| `constraints` | "What should I *not* infer or assume in this domain?" |
| `standards` | "Are there normalization rules specific to this program? (units, categories, naming)" |
| `guidance` | "What non-obvious rule patterns should I look for?" |
| `edge_cases` | "What special populations or situations does this program treat differently?" |

All four sections are asked in sequence. The user can answer "nothing to add" / press Enter to accept proposals as-is or skip a section.

**Step 5: Write `ai-guidance.yaml`**

Compose the file from:
- Goal template base (all fields from the selected goal)
- `source_template: <goal_id>` — added after `goal_id:` field
- `generated_at: <YYYY-MM-DD>` — added after `source_template:`
- Updated `constraints:`, `standards:`, `guidance:` — merged from template + Q&A answers
- `edge_cases:` — new top-level list, populated from Q&A (empty list `[]` if user skipped)

Write to `domains/<domain>/specs/ai-guidance.yaml`.

Print: `Created domains/<domain>/specs/ai-guidance.yaml`

Print next steps:
```
Next: Run /extract-ruleset <domain> to extract the CIVIL ruleset.
      Re-run /refine-guidance <domain> at any time to update guidance.
```

#### Process — UPDATE Mode

**Step 1: Load existing file**

Read `domains/<domain>/specs/ai-guidance.yaml`. Print a summary:
```
Current guidance: <display_name> (source: <source_template>, updated: <generated_at>)
Sections: constraints (<N> items), standards (<N> items), guidance (<N> items), edge_cases (<N> items)
```

**Step 2: Input index check** *(same as CREATE Step 2)*

**Step 3: Doc analysis** *(same as CREATE Step 3, if index available)*

Additionally: compare index topic tags against existing `guidance:` and `edge_cases:` items. Flag any topic areas prominently tagged in the index but not yet mentioned in guidance.

**Step 4: Q&A refinement (targeted)**

If doc analysis flagged gaps, show them first:
```
Potential gaps found in current guidance:
  - [topic area] appears frequently in policy docs but is not covered in guidance or edge_cases
```

Then ask all four section questions in sequence (same as CREATE mode). Prefix each question by showing the existing content for that section:
```
Current [constraints]: (N items)
  - ...
  - ...
What should I *not* infer or assume in this domain? (Enter to keep as-is, or describe changes):
```

Only sections where the user provides input are rewritten. Others are preserved verbatim.

**Step 5: Write updated file**

Overwrite `domains/<domain>/specs/ai-guidance.yaml`. Preserve `source_template`. Update `generated_at` to today.

Print: `Updated domains/<domain>/specs/ai-guidance.yaml`

#### Output table

```
domains/<domain>/specs/ai-guidance.yaml    [CREATED or UPDATED]
```

#### Common Mistakes to Avoid

- Do not add `edge_cases:` to the goal template files in `core/goals/` — they are domain-agnostic; `edge_cases:` belongs only in per-domain `ai-guidance.yaml`
- In UPDATE mode, do not rewrite sections the user did not change — preserve exact wording of unchanged sections
- `source_template` is never updated in UPDATE mode — it records which goal the file was originally created from

```markdown
5. **Load or create `ai-guidance.yaml`?**

   **If it exists:**
   - Read the file

   - **If multiple goal files found:** Present a numbered menu of `display_name` values (same numbered-list style as the file selection prompt above); user selects one
   - Print the selected goal template content for review
   - Ask: `Any domain-specific customizations to add? (Describe changes, or press Enter to use the template as-is.)`
   - Apply any described customizations to the template content
   - Add a `source_template: <goal_id>` field at the top of the file (after `goal_id:`)
   - Write the result to `domains/<domain>/specs/ai-guidance.yaml`
   - Print: `Created domains/<domain>/specs/ai-guidance.yaml`
   - Store the written content for injection

   If the user exits during goal selection or customization, no file is written; the next run will re-prompt.
```

**Replace with:**
```markdown
5. **Load `ai-guidance.yaml`**

   Check for `domains/<domain>/specs/ai-guidance.yaml`:

   **If it exists:**
   - Read the file
   - Print: `Using goal: <display_name> (source: <source_template>)`
   - Store its content for injection in Step 1 (CREATE) or Step 5 (UPDATE)

   **If it does not exist:**
   - Print:
     ```

- [x] `ai-guidance.yaml` written by `/refine-guidance` includes `source_template:`, `generated_at:`, and `edge_cases:` fields

- [x] `/extract-ruleset` stops with a clear error + instructions when `ai-guidance.yaml` is absent
- [x] All pre-flight checks in `/refine-guidance` follow the same structure as other commands

## Dependencies & Risks

**No runtime code changes** — both files are slash command markdown; no transpiler or Python tools are affected.

**Risk: existing domains without `ai-guidance.yaml`** — After this change, anyone running `/extract-ruleset snap` on the current codebase will get an error. Since no `ai-guidance.yaml` files exist yet (confirmed), this is fine — users must run `/refine-guidance` first anyway.

**Risk: goal template compatibility** — The `edge_cases:` field is new and only appears in per-domain files. Existing `core/goals/*.yaml` templates are unchanged. `/extract-ruleset`'s AI Guidance Directive already injects `ai-guidance.yaml` content verbatim, so `edge_cases:` will be included automatically once it exists in the file.
