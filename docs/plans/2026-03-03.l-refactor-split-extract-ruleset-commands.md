# refactor: split extract-ruleset into create/update commands

## Overview

`.claude/commands/extract-ruleset.md` is 842 lines and is loaded in full on every invocation, regardless of whether the user is creating a new ruleset or updating an existing one. Split into two focused commands to reduce per-invocation token cost by ~40–52%.

## Changes

### Files to create

| File | Purpose | Est. size |
|------|---------|-----------|
| `.claude/commands/update-ruleset.md` | UPDATE mode only | ~400 lines |
| `core/civil-authoring-constraints.md` | Compact expression language + `source:` vs `citations:` reference | ~25 lines |

### Files to modify

| File | Change summary |
|------|----------------|
| `.claude/commands/extract-ruleset.md` | Strip to CREATE mode only; add guard; compress pre-flight |

### Size estimates after changes

| File loaded | Current | After |
|------------|---------|-------|
| `extract-ruleset.md` (CREATE run) | 842 lines | ~530 lines |
| `update-ruleset.md` (UPDATE run) | 842 lines | ~400 lines |
| `core/civil-authoring-constraints.md` | — | ~25 lines |

Per-invocation reduction: **~37% (CREATE), ~52% (UPDATE)**

## Implementation Steps

### Step 1 — Create `core/civil-authoring-constraints.md`

New compact reference file (~25 lines). Contains:

- **Expression Language** section (moved from `extract-ruleset.md` lines 139–151):
  - Field access: `Entity.field_name`
  - Constants: `UPPER_SNAKE_CASE`
  - Table lookup: `table('name', key_expr).value_column`
  - Operators: `&&`, `||`, `!`, `==`, `!=`, `<`, `<=`, `>`, `>=`, `+`, `-`, `*`, `/`
  - Functions: `exists(field)`, `is_null(field)`, `between(value, min, max)`, `in(value, [a, b, c])`
  - `computed:` only: `max(a, b)`, `min(a, b)` — computed field names as bare identifiers
  - CIVIL v2 multi-step formulas: use `computed:` for chains, reference final field in `when:`

- **`source:` vs `citations:` distinction** (moved from `extract-ruleset.md` line 165):
  - `source:` = developer traceability (where in the policy doc the element was defined)
  - `citations:` in `add_reason` = legal authority shown to applicants in a denial explanation
  - Same CFR section may appear in both — that is expected and not redundant

- **Footer line:** "For full schema attribute tables, see `core/civil-quickref.md`."

### Step 2 — Modify `extract-ruleset.md` (CREATE mode only)

Apply these changes in order:

#### 2a. Update title and description

Change the subtitle from "Handles both first-time extraction (CREATE) and updating an existing ruleset..." to CREATE-only framing. First sentence: "Create a CIVIL DSL ruleset for a domain from documents in its `input/policy_docs/` subfolder."

#### 2b. Add CIVIL-file-exists guard to Pre-flight

After **check 1** (domain folder exists), add **check 2** (before the current check 2):

```
2. **CIVIL file already exists?**
   - If `domains/<domain>/specs/*.civil.yaml` is present → Print:
       A ruleset already exists for this domain. To update it, run:
         /update-ruleset <domain>
     Then stop.
```

Renumber the remaining checks: 3 → Input docs present, 4 → `<filename>` valid, 5 → Load ai-guidance.yaml, 6 → Multiple input docs.

#### 2c. Remove Mode Detection section

Delete the entire "## Mode Detection" section (lines 99–111). The guard in 2b replaces its CREATE/UPDATE routing logic.

#### 2d. Compress Pre-flight verbatim print blocks

Replace exact quoted terminal output with intent descriptions. Examples:

| Before (verbatim block) | After (intent description) |
|------------------------|---------------------------|
| Full "Domain not found" block (4 lines) | `Print: domain not found, name the missing path, suggest /new-domain <domain>. Stop.` |
| Full "No AI guidance found" block (4 lines) | `Print: no ai-guidance.yaml found, suggest /refine-guidance <domain>. Stop.` |
| Full "Multiple policy documents" display (20 lines) | Keep — this is process-critical content, not a simple error message |

Target: ~20 lines saved in Pre-flight.

#### 2e. Replace CIVIL Reference section

Replace the 35-line "## CIVIL Reference" section with this compact version (~8 lines):

```markdown
## CIVIL Reference

> **Do NOT read `tools/civil_schema.py`, `tools/transpile_to_opa.py`, or any other file in `tools/`
> before authoring any CIVIL YAML.** All syntax needed for authoring is in `core/civil-authoring-constraints.md`.

Read `core/civil-authoring-constraints.md` now before proceeding to Step 4.
```

#### 2f. Rename mode heading

Change `## Process — CREATE Mode` → `## Process`

Remove the "### AI Guidance Directive" wrapper heading. Merge its content into the body of `### Step 1: Read Policy Documents` as an introductory paragraph before the "Identify:" list.

#### 2g. Remove UPDATE Mode section

Delete "## Process — UPDATE Mode" and all its content (lines 529–760). This is the biggest single change (~232 lines).

#### 2h. Update Output table

Remove the "UPDATE" column from the "## Output" table. Keep CREATE column only. Remove the `.stale-cases.yaml` row (UPDATE-only artifact).

Compose the new UPDATE command from existing content. Structure:

```
# Update Ruleset from Changed Policy Documents

Update an existing CIVIL DSL ruleset when input policy documents have changed.

## Input
/update-ruleset <domain>
/update-ruleset <domain> <program>
/update-ruleset <domain> <program> <filename>

## Pre-flight
[Compressed pre-flight — same 5 checks as extract-ruleset.md but with a different guard:]
  NEW check 2: If NO CIVIL file exists → print "No ruleset found. Run /extract-ruleset <domain> first." Stop.

## Scoring Rubric
[Same table as extract-ruleset.md — copy verbatim]

## CIVIL Reference
[Same compact 8-line version as extract-ruleset.md after Step 2e]

## Process
[UPDATE Mode steps 0–9d — moved verbatim from extract-ruleset.md lines 529–760]

## Shared Procedures
[Sub-A, Sub-B, Sub-C — copied verbatim from extract-ruleset.md]

## Output
[Output table with UPDATE column only]

## Common Mistakes to Avoid
[Same 8-item list — copied verbatim]
```

**Guard difference from `extract-ruleset.md`:** Where extract-ruleset stops if a CIVIL file IS found,
update-ruleset stops if a CIVIL file IS NOT found.

## Dependencies & Risks

- **No tool changes** — this is documentation/command refactoring only
- **Existing behavior preserved** — no pipeline logic changes; all steps move verbatim
- **Main risk:** accidentally dropping a step or shared procedure during the split. Mitigation: check line counts and verify all step numbers are present after editing.
- **User habit change:** users running `/extract-ruleset` on existing domains will now get a redirect message instead of running UPDATE. Low risk — the message is clear.
