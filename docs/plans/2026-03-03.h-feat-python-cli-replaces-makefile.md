# feat: Replace Makefile with Python CLI (`x.py`)

## Overview

Replace the current copy-paste-per-domain `Makefile` with a Python script `x.py` and a thin shell shim `./x`. Adding a new domain will require **zero changes** to any existing file — just create the domain directory structure and CIVIL spec.

## Problem Statement

The current `Makefile` requires copying a ~20-line block (5 config vars + 5–6 targets) for every new domain. With multiple modules possible per domain, this is a scaling bottleneck. The Makefile also cannot manage the OPA server lifecycle, so `make <domain>-test` fails if OPA isn't already running.

## Proposed Solution

- **`x.py`** — Python CLI using `argparse`, with `rich` for colored output. Auto-discovers domains and modules from `domains/*/specs/*.civil.yaml`. Manages OPA subprocess for `test` and `pipeline` actions.
- **`./x`** — Thin shell shim (`exec python x.py "$@"`). Users never type `python`.
- **`Makefile`** — Deleted entirely.
- **`requirements.txt`** — Add `rich>=10.0`.

## Interface Design

```
./x <action> [domain] [module]

Actions (domain/module required):
  validate   snap eligibility   Validate CIVIL YAML
  transpile  snap eligibility   Generate Rego from CIVIL
  test       snap eligibility   Start OPA, run tests, stop OPA
  demo       snap eligibility   Start OPA + FastAPI demo (foreground)
  graph      snap eligibility   Generate computation graph
  pipeline   snap eligibility   validate → transpile → test (stops on first failure)

Actions (no domain/module):
  setup                         Install uv, create .venv, install deps, install OPA
  generate-schema               Regenerate core/ruleset.schema.json
  list                          Show all discovered domain/module pairs
```

## Paths Derived by Convention

All paths are derived from `domain` and `module` args — no per-domain config needed:

| Purpose | Path |
|---|---|
| CIVIL spec | `domains/<domain>/specs/<module>.civil.yaml` |
| Rego output | `domains/<domain>/output/<module>.rego` |
| Test cases | `domains/<domain>/specs/tests/<module>_tests.yaml` |
| OPA package | `<domain>.<module>` |
| OPA query path | `/v1/data/<domain>/<module>/decision` |
| Demo script | `domains/<domain>/demo/start.sh` |

## Technical Approach

### `./x` Shell Shim

```sh
#!/bin/sh
exec python x.py "$@"
```

Must be `chmod +x ./x`. Users who haven't activated the venv should run `python x.py setup` directly (system Python) for the first-time bootstrap.

### Bootstrap Note

`rich` is imported at module level. On a fresh clone, `.venv` doesn't exist yet, so `./x setup` cannot import `rich`. **Mitigation**: lazy-import `rich` only in display functions (not at module level), and fall back to plain `print()` if unavailable. This lets `./x setup` work before the venv is created.

```python
def _console():
    try:
        from rich.console import Console
        return Console()
    except ImportError:
        return None  # fallback to plain print
```

### `x.py` Architecture

Single file, structured with `argparse` subparsers. No imports from `tools/` — invoke tools as subprocesses to keep them independent. All paths are resolved relative to `x.py`'s own location, not the working directory.

```python
# x.py — top-level structure

import argparse, subprocess, sys, time, urllib.request, os
from pathlib import Path

ROOT = Path(__file__).parent  # anchor for all paths

def main():
    parser = argparse.ArgumentParser(prog="x", description="xlator CLI")
    sub = parser.add_subparsers(dest="action", required=True)

    # Actions requiring domain + module
    for action in ["validate", "transpile", "test", "demo", "graph", "pipeline"]:
        p = sub.add_parser(action)
        p.add_argument("domain")
        p.add_argument("module")

    # Actions with no domain/module
    sub.add_parser("setup")
    sub.add_parser("generate-schema")
    sub.add_parser("list")

    args = parser.parse_args()
    dispatch(args)
```

### `resolve_paths()` Helper

```python
def resolve_paths(domain, module):
    base = ROOT / "domains" / domain
    return {
        "civil":    base / "specs" / f"{module}.civil.yaml",
        "rego":     base / "output" / f"{module}.rego",
        "tests":    base / "specs" / "tests" / f"{module}_tests.yaml",
        "package":  f"{domain}.{module}",
        "opa_path": f"/v1/data/{domain}/{module}/decision",
        "demo_sh":  base / "demo" / "start.sh",
        "demo_req": base / "demo" / "requirements.txt",
    }
```

### OPA Lifecycle (for `test` and `pipeline`)

`run_tests.py` assumes OPA is already running. `x.py` manages the subprocess:

```python
def start_opa(rego_path, port=8181):
    """Start OPA server, poll health endpoint, return Popen handle."""
    proc = subprocess.Popen(
        ["opa", "run", "--server", "--addr", f":{port}", str(rego_path)],
        stdout=subprocess.DEVNULL, stderr=subprocess.DEVNULL,
    )
    health_url = f"http://localhost:{port}/health"
    for _ in range(10):
        try:
            urllib.request.urlopen(health_url, timeout=1)
            return proc
        except Exception:
            time.sleep(0.5)
    proc.kill()
    raise RuntimeError("OPA failed to start within 5 seconds")

def stop_opa(proc):
    proc.terminate()
    try:
        proc.wait(timeout=5)
    except subprocess.TimeoutExpired:
        proc.kill()
```

Used in `cmd_test()`:
```python
def cmd_test(domain, module):
    paths = resolve_paths(domain, module)
    opa = start_opa(paths["rego"])
    try:
        result = subprocess.run(
            [sys.executable, str(ROOT / "tools" / "run_tests.py"),
             str(paths["tests"]), "--opa-path", paths["opa_path"]],
        )
        sys.exit(result.returncode)
    finally:
        stop_opa(opa)
```

Note: `sys.executable` ensures the same Python that runs `x.py` also runs the tools — no `.venv` activation dependency.

### `./x pipeline` Implementation

`pipeline` calls the internal `cmd_*` functions in sequence (not subprocesses). This avoids spawning an extra Python process per step and gives natural stop-on-failure behavior.

```python
def cmd_pipeline(domain, module):
    cmd_validate(domain, module)   # exits 1 on failure
    cmd_transpile(domain, module)  # exits 1 on failure (also re-validates internally)
    cmd_test(domain, module)       # starts/stops OPA, exits with test returncode
```

### `./x list` Discovery

Uses `ROOT`-anchored glob so it works regardless of working directory:

```python
def cmd_list():
    pattern = str(ROOT / "domains" / "*" / "specs" / "*.civil.yaml")
    rows = []
    for path in sorted(glob.glob(pattern)):
        parts = Path(path).parts
        domain = parts[-4]   # domains/<domain>/specs/<module>.civil.yaml
        module = parts[-1].removesuffix(".civil.yaml")
        rows.append((domain, module))
    # render with rich Table, or plain print if rich not available
```

### `./x setup`

Installs both core deps and demo deps. Uses `shell=True` for `command -v` checks (macOS/bash):

```python
def cmd_setup():
    run("command -v uv || brew install uv", shell=True)
    run("test -d .venv || uv venv", shell=True)
    run(["uv", "pip", "install", "-r", "requirements.txt"])
    # Install demo deps for all domains that have them
    for demo_req in ROOT.glob("domains/*/demo/requirements.txt"):
        run(["uv", "pip", "install", "-r", str(demo_req)])
    run("command -v opa || brew install opa", shell=True)
```

### Error Handling

- Missing file (CIVIL, Rego, tests): print error, exit 1
- OPA fails to start: descriptive error, exit 1
- Subprocess returns non-zero: propagate exit code
- `./x demo` for a domain without `demo/start.sh`: warn and exit 1
- Port 8181 conflict: caught by health-check timeout → "OPA failed to start" message; note in output that port may be in use

## Dependencies & Risks

**Dependencies:**
- `rich>=10.0` — add to `requirements.txt`
- `opa` CLI — must be installed (handled by `./x setup`)

**Risks:**
- **Bootstrap**: `rich` is unavailable before `./x setup`. Mitigated by lazy-importing `rich` with plain-print fallback.
- **OPA port conflict**: Port 8181 in use → `start_opa()` times out → clear error message. No `--port` flag in MVP; document as limitation.
- **`ak_doh` demo**: `domains/ak_doh/demo/` is empty — `./x demo ak_doh apa_adltc` will fail. `x.py` checks for `start.sh` existence and prints a clear error.
- **`transpile_to_opa.py` uses `os.system("python ...")`**: May call wrong Python if venv not activated. Mitigated by `x.py` always calling tools via `sys.executable`. Follow-on: fix `os.system` in transpiler to use `sys.executable`.

## Slash Command Changes

Only `.claude/commands/transpile-and-test.md` references `make`. Two changes needed:

**1. Remove pre-flight step 2 (Makefile check/scaffold)** — the check `grep -n "<domain>-transpile" Makefile` and the entire block that appends a new Makefile entry are obsolete. With `./x`, any domain with a CIVIL spec is automatically supported. Replace step 2 with a simpler file existence check:

```
2. **CIVIL file exists?**
   `domains/<domain>/specs/<module>.civil.yaml` → NO → "CIVIL spec not found." Stop.
```

**2. Replace execution command** — change:
```bash
make <domain>-transpile && make <domain>-test
```
to:
```bash
./x pipeline <domain> <module>
```

The OPA-absent fallback (`python tools/transpile_to_opa.py ...`) remains unchanged — it doesn't use Make.
