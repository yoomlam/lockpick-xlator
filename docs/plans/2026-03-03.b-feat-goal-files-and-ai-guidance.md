# feat: Add Goal Files and `ai-guidance.yaml` to `extract-ruleset`

## Overview

Add a two-layer prompt-engineering system to `/extract-ruleset` that scopes the AI's extraction task before it reads policy documents. Layer 1 is a set of shared, domain-agnostic goal templates (`specs/goals/*.yaml`). Layer 2 is a per-domain instantiation (`domains/<domain>/specs/ai-guidance.yaml`) that is created once from a selected template and reused on all subsequent runs.

## Problem Statement

`/extract-ruleset` currently has no mechanism for the user to tell the AI *what kind* of ruleset to produce before extraction begins. The AI infers structure from the policy document alone, which can lead to inconsistent variable naming conventions, missed intermediate variables, and incorrect output types (e.g., producing a money output when a boolean was needed, or omitting the deduction chain).

The goal files solve this by injecting a scoping directive — role, scope, constraints, standards, guidance, and variable taxonomy — before the policy documents are read.

## Proposed Solution

### Two-layer design

| Layer | File | Scope | Lifecycle |
|---|---|---|---|
| 1 — Templates | `specs/goals/*.yaml` | Shared, domain-agnostic | Created once; maintained by repo maintainers |
| 2 — Instantiation | `domains/<domain>/specs/ai-guidance.yaml` | Per-domain | Created on first run; reused on subsequent runs; user-editable |

### Command flow change

1. User runs `/extract-ruleset <domain>`
2. Pre-flight completes (existing checks unchanged)
3. **New:** Check for `domains/<domain>/specs/ai-guidance.yaml`
   - **Exists** → read it; announce goal in use
   - **Missing** → present goal selection menu → user picks → prompt for customizations → write file
4. Inject `ai-guidance.yaml` content as pre-reading directive
   - CREATE mode: before Step 1 (Read Policy Documents)
   - UPDATE mode: before Step 5 (Re-extract Affected Sections) — *not* Step 1
5. Continue with existing extraction flow

## Technical Approach

### Phase 1: Create `specs/goals/` templates

Create two YAML files. Both share the same schema.

#### Schema (all fields)

```yaml
goal_id: <identifier>            # machine-readable; used in source_template: field of ai-guidance.yaml
display_name: "..."              # shown in selection menu
description: "..."               # one sentence

# Prompt engineering
role: "..."                      # AI's role for this task
scope: "..."                     # focus statement
constraints:
  - "..."                        # hard rules (do not interpret beyond text, cite every rule, etc.)
standards:
  - "..."                        # normalization (units, formats, enums)
guidance:
  - "..."                        # edge cases, conflict resolution, domain-specific reminders

# Variable taxonomy
input_variables:
  categories:
    - category: <name>
      description: "..."
      examples: [field1, field2]

output_variables:
  primary:
    type: "bool"                 # one of: bool, money, float, int (maps to CIVIL decisions[0].type)
    description: "..."
  secondary_decisions:
    - name: "..."
      type: "bool"              # same allowed values as primary: bool, money, float, int
      description: "..."

intermediate_variables:
  categories:
    - category: <name>
      description: "..."
      examples: [field1, field2]

constants_and_tables:
  - name: <name>
    description: "..."

# Examples (CIVIL YAML pseudocode)
examples:
  ruleset_snippet: |
    # ...
  test_case: |
    # ...
```

#### `specs/goals/assess-eligibility.yaml`

```yaml
goal_id: assess-eligibility
display_name: "Assess Program Eligibility"
description: "Determine whether a household or individual qualifies for a benefits program."

role: "You are a policy-to-rules analyst for eligibility determinations."
scope: "Convert the provided policy text into an explicit, testable ruleset for a rules engine."

constraints:
  - "Do not interpret beyond the text; do not add requirements that aren't stated."
  - "Ensure every rule has citations."
  - "Create a list of unknowns/gaps (things needed but not defined in the text)."
  - "List any assumptions made."
  - "Do not invent verification requirements."
  - "Ensure no rule introduces concepts not present in the policy."

standards:
  - "Use monthly income amounts unless policy specifies annual."
  - "Express monetary values in dollars."
  - "Normalize citizenship status as boolean or enum (not free text)."

guidance:
  - "Look for chained deductions: gross → earned deduction → net → shelter deduction → adjusted net income."
  - "Check for both gross and net income tests as separate computed fields."
  - "When federal and state policies conflict, prefer the more restrictive rule."
  - "Deduction amounts typically appear in constants or household-size tables."

input_variables:
  categories:
    - category: household_composition
      description: "Who is in the household — size, relationships, ages."
      examples: [household_size, member_ages, household_type]
    - category: income
      description: "All income sources for the benefit unit."
      examples: [gross_income, earned_income, unearned_income]
    - category: assets
      description: "Countable resources and property."
      examples: [liquid_assets, vehicle_value, bank_balance]
    - category: citizenship_and_residency
      description: "Immigration status and state/program residency requirements."
      examples: [is_citizen, is_resident, immigration_status]
    - category: program_history
      description: "Prior program participation, disqualifications, or time limits."
      examples: [months_on_program, is_disqualified]

output_variables:
  primary:
    type: "bool"
    description: "Binary pass/fail eligibility determination."
  secondary_decisions:
    - name: manual_review_required
      type: "bool"
      description: "Flag for cases requiring human review before determination."

intermediate_variables:
  categories:
    - category: income_deductions
      description: "Allowable deductions applied to gross income."
      examples: [standard_deduction, earned_income_deduction, shelter_deduction]
    - category: income_tests
      description: "Gross and net income values compared against program limits."
      examples: [adjusted_gross_income, net_income, gross_limit, net_limit]
    - category: eligibility_flags
      description: "Boolean sub-decisions that feed the primary eligibility determination."
      examples: [passes_gross_test, passes_net_test, passes_asset_test]

constants_and_tables:
  - name: income_limits
    description: "Gross and net income thresholds by household size."
  - name: standard_deduction
    description: "Fixed deduction amount by household size or program category."
  - name: asset_limits
    description: "Maximum countable resources by eligibility category."

examples:
  ruleset_snippet: |
    # passes_gross_test = (gross_income <= gross_limit)
    # net_income = gross_income - standard_deduction - earned_income_deduction
    # decision = passes_gross_test AND passes_net_test
  test_case: |
    # Given household_size=3, gross_income=$2,100/mo
    # Expect: passes_gross_test=true, passes_net_test=true, decision=true
```

#### `specs/goals/calculate-benefits.yaml`

```yaml
goal_id: calculate-benefits
display_name: "Calculate Benefit Amount"
description: "Derive the dollar amount of benefits a qualifying household receives."

role: "You are a policy-to-rules analyst for benefit amount calculations."
scope: "Convert the provided policy text into explicit, testable benefit calculation rules that produce a dollar amount."

constraints:
  - "Do not interpret beyond the text; do not add requirements that aren't stated."
  - "Ensure every monetary formula traces directly to a policy citation."
  - "Create a list of unknowns/gaps (things needed but not defined in the text)."
  - "List any assumptions made."
  - "Do not round unless the policy explicitly specifies rounding."
  - "Do not apply proration for partial months unless the policy states it."

standards:
  - "Use monthly amounts unless policy specifies otherwise."
  - "Express monetary values in dollars."
  - "Key allotment tables by household size."
  - "Express percentages as decimals (0.3, not 30%)."

guidance:
  - "Maximum allotment is a ceiling from a table — look for it in a table, not a formula."
  - "Check for a minimum benefit floor (do not compute amounts below the minimum)."
  - "Benefit reduction is typically a percentage of net income (e.g., 30% of net income)."
  - "Apply rounding rules only if stated in the policy."
  - "Prorate for partial months only if stated in the policy."

input_variables:
  categories:
    - category: household_composition
      description: "Who is in the household — size for table lookups."
      examples: [household_size, member_count]
    - category: income
      description: "Net income used in benefit calculation formula."
      examples: [net_income, adjusted_net_income]
    - category: expenses
      description: "Out-of-pocket costs that may affect benefit formula."
      examples: [shelter_costs, utility_costs]
    - category: benefit_history
      description: "Prior benefit amounts or participation flags relevant to calculation."
      examples: [prior_benefit_amount, is_new_applicant]

output_variables:
  primary:
    type: "money"
    description: "Dollar amount of benefits the qualifying household receives."
  secondary_decisions:
    - name: receives_minimum_benefit
      type: "bool"
      description: "Flag indicating the household receives the minimum benefit floor."

intermediate_variables:
  categories:
    - category: maximum_allotment
      description: "Table-derived ceiling for benefit amount by household size."
      examples: [maximum_allotment, thrifty_food_plan_amount]
    - category: benefit_reduction
      description: "Calculated reduction applied to the maximum allotment."
      examples: [income_reduction, net_income_share]
    - category: proration
      description: "Partial-month or partial-period adjustment factors."
      examples: [proration_factor, days_in_period]

constants_and_tables:
  - name: maximum_allotment
    description: "Maximum monthly benefit by household size."
  - name: minimum_benefit
    description: "Minimum monthly benefit floor (if applicable)."
  - name: income_reduction_rate
    description: "Percentage of net income used to reduce the maximum allotment."

examples:
  ruleset_snippet: |
    # benefit_amount = max(maximum_allotment - (net_income * 0.30), minimum_benefit)
    # maximum_allotment is from a table keyed by household_size
  test_case: |
    # Given household_size=3, net_income=$500/mo, maximum_allotment=$658
    # Expect: benefit_amount = max(658 - 150, 23) = $508
```

### Phase 2: Modify `.claude/commands/extract-ruleset.md`

#### 2a. Pre-flight addition (after existing checks, before mode detection)

Insert after the existing pre-flight checks, before mode detection. Treat as a **new Pre-flight Step 5** (existing steps 1-4 are: domain folder check, input docs check, filename check, file selection prompt).

```
Pre-flight Step 5: Load or create ai-guidance.yaml

Check for `domains/<domain>/specs/ai-guidance.yaml`:

IF it exists:
  - Read the file
  - Print: "Using goal: <display_name> (source: <source_template>)"
  - Store content as `$guidance`

IF it does not exist:
  - Scan `specs/goals/*.yaml` for all goal files
  - IF no goal files found:
      Print: "No goal templates found in specs/goals/. Proceeding without guidance."
      Set $guidance = nil; continue
  - IF exactly one goal file found:
      Print: "One goal template found: <display_name>. Using it."
      Confirm: "Continue with this goal? [y/n]"
      If n: exit
  - IF multiple goal files found:
      Present numbered menu of display_name values (same style as file selection menu)
      User selects a goal
  - Print the selected goal template content
  - Ask: "Any domain-specific customizations to add?
          (Describe changes, or press enter to use the template as-is.)"
  - Apply described customizations to the template
  - Add `source_template: <goal_id>` field to the file
  - Write to `domains/<domain>/specs/ai-guidance.yaml`
  - Print: "Created domains/<domain>/specs/ai-guidance.yaml"
  - Store content as `$guidance`
```

**Edge case:** User exits during goal selection or customization → no file is written; next run re-prompts.

#### 2b. Injection in CREATE mode (before Step 1)

Insert before CREATE Step 1 (Read Policy Documents):

```
IF $guidance is set:
  Before reading any policy documents, internalize the following extraction goal:

  ---
  [paste ai-guidance.yaml content as a fenced yaml block]
  ---

  Use this goal to scope your reading: prioritize sections relevant to
  <input_variables categories>, watch for <intermediate_variables categories>,
  and target a <output_variables.primary.type> output called `decision`.
  Apply all constraints and standards listed above throughout this extraction.
```

#### 2c. Injection in UPDATE mode (before Step 5, not Step 1)

UPDATE Step 1 (Load Baseline) and Steps 2-4 (change detection) are unaffected. The directive is injected immediately before Step 5 (Re-extract Affected Sections):

```
IF $guidance is set:
  Before re-reading affected policy sections, recall the extraction goal:

  ---
  [paste ai-guidance.yaml content as fenced yaml block]
  ---

  Apply these constraints and standards when re-extracting the affected CIVIL sections.
```

#### 2d. Output table update

Add row to the output table at the bottom of the command:

| File | CREATE | UPDATE |
|---|---|---|
| `domains/<domain>/specs/ai-guidance.yaml` | Created (first run) / Read-only (subsequent) | Read-only |

## Dependencies & Risks

- `extract-ruleset.md` is 700 lines — the pre-flight addition and two injection points must slot precisely to avoid disrupting numbered step references used elsewhere in the command
- The customization step relies on Claude applying free-form user descriptions to a YAML template — no validation of the resulting `ai-guidance.yaml` is performed; malformed YAML is an error risk (acceptable for now; validation can be added later)
- Multi-program domains (`ak_doh`) use one `ai-guidance.yaml` for all programs — if programs within a domain have different goal types, the user must manually edit the file or maintain multiple versions (out of scope for this plan)
- By design, `ai-guidance.yaml` uses category descriptions and examples (not frozen field names), so it does not duplicate `naming-manifest.yaml`; the separation is enforced by the schema, not by runtime checks
