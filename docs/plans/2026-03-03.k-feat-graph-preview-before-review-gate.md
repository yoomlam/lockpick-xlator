# feat: Computation Graph Preview Before Human Review Gate

## Overview

Two coordinated changes to `.claude/commands/extract-ruleset.md`:

1. **Graph preview step (Step 6b / Step 8b):** Generate the computation graph *before* the Human Review Gate in both CREATE and UPDATE modes. Surface the graph inline at the top of the gate so the user has a dependency map while reviewing rules.
2. **Shared Subroutines refactor:** Extract three verbatim-duplicated blocks (Validate, Generate Computation Graph, Extraction Complete Footer) into a `## Shared Procedures` section, eliminating word-for-word duplication between CREATE and UPDATE modes.

## Problem Statement

The current flow generates the computation graph *after* user approval (Step 7c / Step 9d). Users reviewing complex rulesets — many computed fields, multi-step deduction chains — have no structural overview during review. They cannot easily catch a missing dependency or a field wired to the wrong rule.

Additionally, `extract-ruleset.md` is 763 lines with three blocks duplicated verbatim across CREATE and UPDATE modes. Any future edit to Validate logic, graph generation, or the closing message must be applied in two places.

## Proposed Solution

### Change 1: Graph Preview Steps (6b and 8b)

Insert between Validate and the Human Review Gate in both modes:

```
Step 6b (CREATE) / Step 8b (UPDATE): Generate Computation Graph (Preview)
```

**Behavior:**

1. Run `python tools/computation_graph.py domains/<domain>/specs/<program>.civil.yaml`
   Always run unconditionally — regenerates even if graph files already exist from a prior run.
   Capture stdout. Do not echo verbatim.

2. **On success (exit 0):**
   Read `domains/<domain>/specs/<program>.graph.yaml`. Extract all nodes where `kind == "computed"`.

3. **On failure (exit 1):**
   Print `Warning: computation graph preview could not be generated — continuing to review.`
   Proceed to the gate without showing graph content.

**At the Human Review Gate**, if Step 6b/8b succeeded, show the following block **before** the `Review summary:` header (CREATE) or **before** the `Changed input docs:` block (UPDATE):

```
✓ Draft graph: domains/<domain>/specs/<program>.graph.md

Dependency graph (computed fields):
  <field_1>    ← <depends_on_1>, <depends_on_2>    → <used_by_1>
  <field_2>    ← <depends_on_1>                    → [rule: <rule_id>]
  ...

```mermaid
[contents of <program>.graph.md — the flowchart LR block]
```
```

**Zero computed fields:** If no `kind: computed` nodes exist in `.graph.yaml`, replace the dependency table with `(No computed fields in this module.)`. Still show the Mermaid block (it may contain direct input→rule edges).

### Change 2: Shared Procedures Section

Add `## Shared Procedures` after `## Process — UPDATE Mode` and before `## Output`. Contain three named subroutines:

**Sub-A: Validate**
- Full retry loop (same text as current CREATE Step 6)
- Both CREATE Step 6 and UPDATE Step 8 replaced with: `Run **Sub-A: Validate**.`

**Sub-B: Generate Computation Graph**
- Bash command + failure warning
- Failure message distinguishes stale-graph risk: *"Warning: computation graph could not be refreshed. The draft graph at `<path>.graph.md` may reflect pre-approval state."*
- Both CREATE Step 7c and UPDATE Step 9d replaced with: `Run **Sub-B: Generate Computation Graph**.`

**Sub-C: Extraction Complete Footer**
- "Extraction complete." message
- Conditional "Note: other policy docs" block
- "Next steps" block
- Both CREATE and UPDATE closing blocks replaced with: `Run **Sub-C: Extraction Complete Footer**.`

**Note on 6b/8b:** The graph preview steps stay *inline* (not a subroutine) because they also drive what gets surfaced at the gate — the surrounding context (full-review vs. diff-review) is mode-specific enough that extracting them would require a parameter and add more complexity than it removes.

## Technical Considerations

### Dependency summary rendering
The command reads `.graph.yaml` (written by the tool). Format for each computed node:
```
<node_key>    ← <depends_on[0]>, <depends_on[1]>    → <used_by[0]>, [rule: <used_by[1]>]
```
- `depends_on` list: join with `, ` — raw field names, no decoration
- `used_by` list: prefix rule-kind nodes with `[rule: ]`; plain field names for computed-kind nodes
- If `depends_on` is empty: show `← [no deps]`
- If `used_by` is empty: show `→ [unused]` (dead-code hint)

### Mermaid block
Read the full content of `<program>.graph.md`. Extract the fenced `mermaid` block (between ` ```mermaid ` and ` ``` `). Embed inline. Do not re-generate — use exactly what the tool wrote.

### graph.yaml node structure (for reference)
```yaml
<node_key>:
  kind: input | computed | rule
  type: money | bool | float | int | ...
  location: "<yaml path>"
  depends_on: [<ref>, ...]
  used_by: [<ref>, ...]
```
Filter for `kind: computed` to build the dependency summary.

### `computation_graph.py` CLI
```bash
python tools/computation_graph.py <civil_yaml>
# exit 0: success — both .graph.yaml and .graph.md written
# exit 1: any error (file not found, YAML parse, expression parse)
```
Stdout on success prints two file paths (not needed — derive paths from convention).

### No precedent for Shared Procedures
No other command file uses this pattern. The new section is a first. Keep the subroutine names short and prefixed (`Sub-A`, `Sub-B`, `Sub-C`) so they are visually distinct from regular step headers.
