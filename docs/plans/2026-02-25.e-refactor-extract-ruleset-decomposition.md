# refactor: Decompose extract-ruleset into Three Focused Slash Commands

## Overview

Split `.claude/commands/extract-ruleset.md` (~680 lines) into three focused slash commands by removing test authoring and transpile/test steps into dedicated commands. The goal is to reduce context window usage for each command invocation and apply single responsibility principle.

## Problem Statement / Motivation

`/extract-ruleset` currently carries three distinct responsibilities in ~680 lines:
1. CIVIL extraction, validation, and human review
2. Test case authoring and maintenance
3. Transpilation and runtime test execution

Every invocation loads the entire file into context, including steps that won't run in that session (e.g., test steps load even during extraction). Splitting by concern reduces per-invocation context by ~150–200 lines and makes each command independently re-runnable.

## Proposed Solution

Three commands, each with a single concern:

| Command | Responsibility | Mode-aware? | Standalone? |
|---------|----------------|-------------|-------------|
| `/extract-ruleset` (trimmed) | CIVIL extraction + review gate | Yes (CREATE/UPDATE) | Yes |
| `/create-tests` (new) | Test file creation and maintenance | Yes (CREATE/UPDATE) | Yes |
| `/transpile-and-test` (new) | Transpile Rego + run test suite | No | Yes |

### Handoff flow
After the review gate, `/extract-ruleset` prints:
```
Extraction complete.
Next steps:
  1. /create-tests <domain>
  2. /transpile-and-test <domain>
```
No prompt — the command is done. User runs the next commands when ready.

### Stale case sidecar
When `/extract-ruleset` runs in UPDATE mode and its review gate passes, it writes `domains/<domain>/specs/.stale-cases.yaml` listing test cases whose boundary inputs match old table/constant values. `/create-tests` reads this file if present (post-extract-ruleset flow) or derives stale cases independently via table/constant comparison (standalone flow). `/create-tests` deletes the sidecar after consuming it.

### create-tests.md (new file)

**Input:** `/create-tests [<domain>]` or `/create-tests [<domain> <program>]`
- If `<domain>` omitted: list `domains/*/` and prompt
- If multiple `*.civil.yaml` in `core/` and `<program>` omitted: prompt to choose

**Pre-flight:**
- Verify `domains/<domain>/` exists
- Verify `domains/<domain>/specs/<program>.civil.yaml` exists; if not: stop with "No CIVIL file found. Run `/extract-ruleset <domain>` first."
- Create `domains/<domain>/specs/tests/` silently if absent

**Mode detection:**
```bash
ls domains/<domain>/specs/tests/<program>_tests.yaml 2>/dev/null
```
File absent → CREATE mode. File present → UPDATE mode.

**CREATE mode:**
- Read the CIVIL file: extract facts, rules, computed fields, tables, constants
- Draft at minimum 6 test cases with these tags:
  - `allow` — all conditions comfortably met (happy path)
  - `deny` + gross test — fails gross income threshold (if one exists)
  - `deny` + net test — passes gross, fails net after deductions (if net check exists)
  - `allow` + exemption — elderly, disabled, or other exemption path (if one exists)
  - `allow` + boundary — income or value exactly at a threshold (≤ limit = pass)
  - `deny` + edge — large household (size 9+), all-zero income, or other extreme
- Inputs must be flat key-value (never nested by entity name)
- Write `domains/<domain>/specs/tests/<program>_tests.yaml`
- Reference format: `domains/snap/specs/tests/eligibility_tests.yaml`

**UPDATE mode:**
1. Check for `domains/<domain>/specs/.stale-cases.yaml`:
   - If present: load stale case list from it
   - If absent: compare each test case's `inputs` values against all current `tables:` rows and `constants:` values; flag cases where any input matches a value no longer present
2. For each stale case: update threshold values in `inputs` and `expected` to match new CIVIL values
3. Add new test cases for any facts, rules, or computed fields added in this update (maintain 6-tag coverage)
4. Write updated `domains/<domain>/specs/tests/<program>_tests.yaml`
5. Delete `domains/<domain>/specs/.stale-cases.yaml` after consuming it (prevent stale context on next run)

**Edge cases:**
- No test cases identified as stale (UPDATE standalone): print "No stale cases detected from table/constant comparison. Review manually for logic-only rule changes." then proceed to add new coverage

### transpile-and-test.md (new file)

**Input:** `/transpile-and-test [<domain>]`
- If `<domain>` omitted: list `domains/*/` and prompt

**Pre-flight:**
- Verify `domains/<domain>/` exists
- Verify Makefile has `<domain>-transpile` target; if absent: warn "Run `/extract-ruleset <domain>` first to scaffold the Makefile target" and stop
- Check for OPA in PATH; if absent: warn "OPA not found. Transpilation will run but tests will be skipped. Install OPA to run the full suite."

**Execution (OPA present):**
```bash
make <domain>-transpile && make <domain>-test
```
Relay make output verbatim. No summary formatting.

**Execution (OPA absent — fallback):**
```bash
python tools/transpile_to_opa.py \
    domains/<domain>/specs/<program>.civil.yaml \
    domains/<domain>/output/<program>.rego \
    --package <domain>.<program>
```
Print: "Transpilation complete. Tests skipped — OPA not in PATH."

**On test failure:**
Show failing case IDs and actual vs. expected values. Ask the user to diagnose:
- **Rule error** — the CIVIL `when:` expression is wrong → fix in CIVIL file, re-run `/extract-ruleset`
- **Test expectation error** — the test case has wrong expected values → fix in tests file, re-run `/transpile-and-test`
- **Transpiler bug** — Rego generation is incorrect → file a transpiler issue; do not modify CIVIL or tests

## Dependencies & Risks

**No runtime code changes.** All three files are prompt documents for Claude; no Python, Rego, or Makefile changes required.

**Risk: step renumbering confusion.** After deletions, CREATE steps jump from 6 to 7 (was 8) and UPDATE steps from 8 to 9 (was 10). Verify numbering is sequential in the final file.

**Risk: stale-cases sidecar left on disk.** If `/extract-ruleset` writes `.stale-cases.yaml` but the user never runs `/create-tests`, the file persists. This is acceptable — the file is tiny and prefixed with `.` (hidden). Document in `/create-tests` that it is safe to delete manually if the full pipeline is not run.

**Risk: `.stale-cases.yaml` format not yet defined.** The format needs to be specified in the commands themselves. Proposed minimal format:
```yaml
# Written by /extract-ruleset UPDATE mode. Consumed by /create-tests.
stale_cases:
  - case_id: "deny_gross_003"
    reason: "gross_limit threshold changed from 2311 to 2430"
  - case_id: "allow_boundary_001"
    reason: "net_limit for household_size 3 changed from 1768 to 1850"
```
