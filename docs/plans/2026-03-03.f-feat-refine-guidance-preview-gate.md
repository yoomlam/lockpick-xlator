# feat: Add Rule Preview Gate to /refine-guidance

## Overview

Insert a new **Step 5: Preview Gate** into both CREATE and UPDATE modes of `.claude/commands/refine-guidance.md`. `ai-guidance.yaml` is written incrementally during Step 4 — after each section's Q&A answer — so the file is fully current when Step 5 opens. The preview gate then presents 2–3 sample CIVIL rules alongside the live file, and loops through a confirm/refine cycle (updating the file on each refinement) until the user explicitly confirms.

## Problem Statement / Motivation

Currently, `ai-guidance.yaml` is written once at the end, after Step 4 Q&A completes. Users have no way to verify that their guidance answers will produce the right extractions until they run `/extract-ruleset` — a much heavier operation. The preview gate closes this feedback gap: by writing the file incrementally during Step 4, the user arrives at Step 5 with a live, complete file they can open alongside the CIVIL samples to make a more informed assessment.

## Proposed Solution

Add a new Step 5 (shifting current Step 5 write → Step 6) in both CREATE and UPDATE modes. Step 4 gains incremental file writes. Step 5 draws from Step 3 observations, presents illustrative CIVIL snippets with source quotes, and loops back to targeted Q&A refinement (updating the file each time) until the user explicitly confirms.

```
Step 4: Q&A (each section answer → write/update ai-guidance.yaml)
       ↓
[Step 5: Preview Gate]
  ai-guidance.yaml is already current — user can open it
  Show 2–3 sample rules + source text
  Ask: looks good or refine?
       ↓                       ↓
  [a] Looks good           [1–4] Refine section
       ↓                       ↓
  Step 6: Done msg      Re-ask that section's Q&A
                        (pre-fill current answer)
                             ↓
                        Update ai-guidance.yaml
                             ↓
                        Return to Preview Gate
                        (regenerate preview)

  [x] Reset → rollback file, restart Step 4
  [q] Quit → file stays as-is, exit
```

## Technical Considerations

### Incremental writes during Step 4

After each section's Q&A answer in Step 4 (constraints, standards, guidance, edge_cases), write/update `ai-guidance.yaml` immediately — before asking the next section's question. By the time Step 4 completes, the file is fully populated and ready to reference.

In back-navigation loops (Step 5 → refine section → return), the same rule applies: write/update the file immediately after the user answers the re-asked section question, before regenerating the preview.

**CREATE mode:** The file is created on the first section answer and overwritten on each subsequent one.

**UPDATE mode:** Before Step 4 begins, capture the current file contents in memory as a rollback snapshot (it was already loaded in Step 1). Overwrite the file after each section answer as in CREATE mode.

### Preview generation algorithm

The preview is synthesized by the AI using observations already in context from Step 3 — specifically topic clusters, section headings, and policy excerpts gathered from `input-index.yaml`. The assistant maps prominent topic areas to CIVIL constructs (rules, computed, tables) and selects 2–3 representative examples spanning different rule types where possible.

**Source text:** Each preview entry quotes a sentence or short passage from the `input-index.yaml` section summary (not the raw policy doc). This is always available if Step 3 ran.

**Fallback when Step 3 was skipped** (no `input-index.yaml`): The preview gate still appears. The assistant synthesizes illustrative examples from the Q&A answers and goal template alone, but omits the "Source:" line and adds a note:
> *(No input index available — examples are illustrative only. Run `/index-inputs <domain>` for source-grounded previews.)*

**Fallback when context has faded** (large docs): Re-read `domains/<domain>/specs/input-index.yaml` silently — it is a small structured file that reliably reconstructs Step 3 signals without re-ingesting full policy docs.

### Back-navigation pre-fill

When the user picks a section to refine, show the **most recent answer for that section** as the pre-filled default — not the original Step 3 proposals. On a second+ loop, this means showing the user's prior edited value, not the first-pass proposals. The Q&A format is:

```
Current [constraints]: (N items)
  - "Do not interpret beyond the text..."
  - [user's previous answer...]
What should I *not* infer or assume in this domain? (Enter to keep as-is):
```

After the user answers, return **immediately** to the Preview Gate (do not continue through the remaining sections). Cross-section staleness is a known limitation — the user can refine multiple sections by looping through the gate repeatedly.

### [x] Reset scope

"Start over" resets Q&A answers and rolls back `ai-guidance.yaml`:

- **CREATE mode:** Delete `domains/<domain>/specs/ai-guidance.yaml`, then restart Step 4.
- **UPDATE mode:** Restore the file to the rollback snapshot captured at the start of Step 4, then restart Step 4.

Goal selection (Step 1), index work (Step 2), and doc analysis (Step 3) are preserved in both modes.

### [q] Quit behavior

`[q]` exits the command without further changes. Because `ai-guidance.yaml` has been written incrementally, the file is left in its current state — it is not deleted or rolled back. Print:
> `Exiting. ai-guidance.yaml saved with current answers. Run /refine-guidance <domain> to continue refining.`

### Free-form input handling

If the user sends text that doesn't match [a], [1]–[4], [x], or [q], re-display the gate options and re-prompt. Do not attempt to interpret free-form text as a refinement.

### UPDATE mode behavior

In UPDATE mode, the preview should reference the **existing guidance structure** (already loaded in Step 1) rather than purely synthetic examples. If CIVIL rules already exist in `domains/<domain>/specs/`, note this in the preview header:
> *(Showing illustrative samples based on updated guidance — existing rules will be preserved in the merge.)*

### Step numbering

Both CREATE and UPDATE modes have 5 steps currently. After this change, both will have 6. Rename all "Step 5" headers to "Step 6".

The UPDATE mode has two prose cross-references (`*(same as CREATE Step 2)*` and `*(same as CREATE Step 3, if index available)*`) — these reference Steps 2 and 3, which are unaffected by this change. No renaming needed there.

## Dependencies & Risks

- **No code changes.** Pure markdown edit to one command spec file.
- **Preview quality is illustrative, not validated.** CIVIL snippets are synthesized — they may not pass `validate_civil.py`. The caveat line addresses this.
- **Context window risk** is mitigated by the `input-index.yaml` fallback.
- **Cross-section staleness** (refining section 2 after answering section 4) is a known limitation, not a bug. Users can loop back to any section independently.
- **Partial file state on [q].** If a user quits mid-Step-4 (after answering 2 of 4 sections), the file exists but is incomplete. This is acceptable — the file is always a valid YAML subset, and the user is told where it is. Running `/refine-guidance` again starts UPDATE mode and completes the remaining sections.
- **UPDATE mode rollback** requires the snapshot captured in memory at Step 4 start. This snapshot is already in context (Step 1 loads the full file). No additional file read is needed.
