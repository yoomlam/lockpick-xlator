# feat: Add Pydantic Schema for CIVIL DSL Validation

## Overview

Replace the documentation-only `core/ruleset_schema.yaml` (377 lines of pure `#` comments, yields `None` when loaded) and the hardcoded Python constants in `tools/validate_civil.py` with a single Pydantic v2 model file (`tools/civil_schema.py`). Pydantic v2's `model_json_schema()` generates `core/ruleset.schema.json` automatically, giving VS Code YAML autocompletion for every `.civil.yaml` file. The validator is rewritten to call `CivilModule.model_validate()` instead of manual dict-walking.

**Deliverables:**
- `tools/civil_schema.py` — Pydantic v2 models (new, source of truth)
- `tools/validate_civil.py` — rewritten to use Pydantic
- `core/ruleset.schema.json` — generated artifact, committed
- `.vscode/settings.json` — wires editor autocompletion
- `requirements.txt` (top-level) — adds `pydantic>=2.0` for tools
- `Makefile` — adds `generate-schema` target

**Unchanged:** `tools/transpile_to_opa.py`, `tools/run_tests.py`, `core/ruleset_schema.yaml` (kept as narrative supplement).

---

## Problem Statement

Three problems exist today:

1. **Dual source of truth.** Valid enum values (`VALID_PRIMITIVE_TYPES`, `VALID_PRECEDENCES`, etc.) are hardcoded in `tools/validate_civil.py` as Python constants. Those same values are also documented in `core/ruleset_schema.yaml` as comments. Adding a new type (e.g., `decimal`) requires editing both files — and they can drift silently.

2. **No editor support.** VS Code has no schema awareness for `.civil.yaml` files. No autocompletion, no hover docs, no inline errors.

3. **Validator is structural.** `validate_civil.py` is ~220 lines of manual `isinstance()` / `.get()` / `errors.append()` chains. It reimplements what a typed model library gives for free, and it silently skips `tables:` and `constants:` sections entirely.

---

## Proposed Solution

**Pydantic v2 models** (`tools/civil_schema.py`) become the single source of truth for the CIVIL DSL structure. The validator delegates structural checks to `CivilModule.model_validate(data)` and keeps only the two checks that Pydantic cannot express natively:

1. Expression-language presence check (`when:` and `expr:` must be non-empty strings — already enforced by Pydantic's `str` type + `min_length=1` if desired)
2. Rule `id` uniqueness across the `rules:` list (a `@model_validator` on `CivilModule`)

`core/ruleset.schema.json` is generated by running `python tools/civil_schema.py` (a `__main__` block) and committed to the repo. VS Code's YAML extension reads it automatically via `.vscode/settings.json`.

### Phase 1: `tools/civil_schema.py` — Pydantic Model Hierarchy

Build models bottom-up (leaf → root):

#### 1a. Leaf models

```python
# tools/civil_schema.py

from __future__ import annotations
from typing import Any, Literal
from pydantic import BaseModel, ConfigDict, Field, field_validator, model_validator
from annotated_types import Ge, Le
from typing import Annotated

ScoreField = Annotated[int, Field(ge=1, le=5)]

class ReviewBlock(BaseModel):
    extraction_fidelity: ScoreField
    source_clarity: ScoreField
    logic_complexity: ScoreField
    policy_complexity: ScoreField
    notes: str | None = None

class Citation(BaseModel):
    label: str
    url: str | None = None
    excerpt: str | None = None

class Conditional(BaseModel):
    """Wraps CIVIL if/then/else — avoids Python keyword collision via aliases."""
    model_config = ConfigDict(populate_by_name=True)
    if_: str = Field(alias="if")
    then: str
    else_: str = Field(alias="else")
```

#### 1b. Facts, decisions, tables, constants

```python
PRIMITIVE_TYPES = {"int", "float", "bool", "string", "date", "money", "list", "set", "enum"}

class FactField(BaseModel):
    type: str
    description: str | None = None
    optional: bool = False
    currency: str | None = None
    values: list[str] | None = None   # enum fact fields

    @field_validator("type")
    @classmethod
    def warn_unknown_type(cls, v: str) -> str:
        if v not in PRIMITIVE_TYPES:
            import sys
            print(f"WARNING: unknown fact field type '{v}' (may be a custom type)", file=sys.stderr)
        return v

class FactEntity(BaseModel):
    description: str | None = None
    fields: dict[str, FactField]

class DecisionField(BaseModel):
    type: str
    default: Any = None
    description: str | None = None
    item: str | None = None   # for list/set types

class TableDef(BaseModel):
    description: str | None = None
    key: list[str]
    value: list[str]
    rows: list[dict[str, Any]]
```

#### 1c. Computed fields

```python
COMPUTED_TYPES = {"money", "bool", "float", "int"}

class ComputedField(BaseModel):
    type: Literal["money", "bool", "float", "int"]
    currency: str | None = None
    description: str | None = None
    expr: str | None = None
    conditional: Conditional | None = None
    review: ReviewBlock | None = None

    @model_validator(mode="after")
    def expr_xor_conditional(self) -> "ComputedField":
        has_expr = self.expr is not None
        has_cond = self.conditional is not None
        if has_expr == has_cond:   # both or neither
            raise ValueError("ComputedField must have exactly one of 'expr' or 'conditional'")
        return self
```

#### 1d. Rules and actions

```python
class AddReasonContent(BaseModel):
    code: str
    message: str
    citations: list[Citation] = []

class AddInstructionContent(BaseModel):
    step: str
    message: str
    citations: list[Citation] = []

class Action(BaseModel):
    """Each THEN action is a dict with exactly one action-type key."""
    add_reason: AddReasonContent | None = None
    set: dict[str, Any] | None = None
    add_instruction: AddInstructionContent | None = None
    add_to_set: dict[str, Any] | None = None
    append_to_list: dict[str, Any] | None = None

    @model_validator(mode="before")
    @classmethod
    def one_action_type(cls, data: Any) -> Any:
        if isinstance(data, dict):
            known = {"add_reason", "set", "add_instruction", "add_to_set", "append_to_list"}
            present = [k for k in data if k in known]
            if len(present) != 1:
                raise ValueError(f"Each 'then' action must have exactly one type; got {present}")
        return data

class Rule(BaseModel):
    id: str
    kind: Literal["deny", "allow"]
    priority: int
    when: str
    then: list[Action]
    description: str | None = None
    review: ReviewBlock | None = None
```

#### 1e. Top-level module (root)

```python
class Jurisdiction(BaseModel):
    level: Literal["federal", "state", "county", "city"]
    country: str
    state: str | None = None
    county: str | None = None
    city: str | None = None

class Effective(BaseModel):
    start: Any   # date string — no strict format enforcement (matches current validator)
    end: Any = None

class RuleSet(BaseModel):
    name: str
    precedence: Literal[
        "deny_overrides_allow", "allow_overrides_deny",
        "first_match", "priority_order"
    ] | None = None
    description: str | None = None

class CivilModule(BaseModel):
    module: str
    description: str
    version: str
    jurisdiction: Jurisdiction
    effective: Effective
    facts: dict[str, FactEntity]
    decisions: dict[str, DecisionField]
    rule_set: RuleSet
    rules: list[Rule]
    # optional sections
    tables: dict[str, TableDef] | None = None
    constants: dict[str, Any] | None = None
    computed: dict[str, ComputedField] | None = None
    types: dict[str, Any] | None = None   # custom type defs — not validated structurally

    @model_validator(mode="after")
    def unique_rule_ids(self) -> "CivilModule":
        seen: set[str] = set()
        for rule in self.rules:
            if rule.id in seen:
                raise ValueError(f"Duplicate rule id: '{rule.id}'")
            seen.add(rule.id)
        return self
```

#### 1f. `__main__` block for JSON Schema generation

```python
# tools/civil_schema.py (bottom of file)

if __name__ == "__main__":
    import json, pathlib
    schema = CivilModule.model_json_schema()
    out = pathlib.Path("core/ruleset.schema.json")
    out.write_text(json.dumps(schema, indent=2) + "\n")
    print(f"Generated {out}")
```

### Phase 2: Rewrite `tools/validate_civil.py`

Replace ~220 lines of manual validation with:

```python
# tools/validate_civil.py

import sys
import yaml
from pydantic import ValidationError
# civil_schema.py is in the same directory; add tools/ to path if needed
import pathlib
sys.path.insert(0, str(pathlib.Path(__file__).parent))
from civil_schema import CivilModule


def validate(path: str) -> bool:
    with open(path) as f:
        data = yaml.safe_load(f)

    try:
        CivilModule.model_validate(data)
        return True
    except ValidationError as e:
        for err in e.errors():
            loc = " → ".join(str(x) for x in err["loc"]) if err["loc"] else "(root)"
            print(f"ERROR: {loc}: {err['msg']}", file=sys.stderr)
        print(f"\n{e.error_count()} error(s) found in {path}", file=sys.stderr)
        return False


if __name__ == "__main__":
    if len(sys.argv) != 2:
        print("Usage: python tools/validate_civil.py <path_to_civil_yaml>", file=sys.stderr)
        sys.exit(1)
    ok = validate(sys.argv[1])
    sys.exit(0 if ok else 1)
```

**CLI compatibility preserved:** same single-argument interface, same exit codes (0 = valid, 1 = invalid), same `ERROR:` prefix on stderr. The `extract-ruleset.md` command's call at step 6 works unchanged.

## Alternative Approaches Considered

See [brainstorm document](../brainstorms/2026-02-25-civil-schema-pydantic-brainstorm.md) for full analysis. Summary:

- **JSON Schema (hand-written)** — Standard but verbose JSON to maintain; still needs separate Python logic.
- **Structured YAML + generator** — YAML stays authoritative but adds a non-standard build step.
- **Pydantic (chosen)** — Python is the project language; JSON Schema generates automatically; validator simplifies dramatically.

## Risk Analysis

| Risk | Mitigation |
|------|-----------|
| Pydantic error messages differ from current format | Verify format with test cases before finalizing; adjust `loc` formatting in `validate()` if needed |
| Pydantic `Conditional` aliases break YAML loading | Test with `eligibility.civil.yaml` which uses `conditional:` with `if`/`then`/`else` keys |
| `tables:`/`constants:` validation rejects currently-valid files | Keep `tables` and `constants` models permissive; `TableDef` and `dict[str, Any]` are not strict |
| JSON Schema has different null/optional semantics | Validate that VS Code shows correct completions for optional fields; adjust `Optional` vs default handling |
