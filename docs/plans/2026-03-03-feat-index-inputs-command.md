# feat: Add `/index-inputs` Slash Command

## Overview

Create a new slash command `/index-inputs` that scans a domain's `input/policy_docs/` directory and generates `specs/input-index.yaml` — a structured index of all policy document sections with LLM-generated summaries and topic tags. On first run (CREATE), all files are processed. On subsequent runs (UPDATE), only changed or new files are re-indexed using git SHA comparison, preserving existing index entries for unchanged files.

## Problem Statement / Motivation

The `ak_doh` domain now contains 8+ policy documents spread across nested subdirectories (`apa_manual/441/`, `apa_manual/442/`). When `/extract-ruleset` or a human needs to find "which document covers income exclusions," there's no structured index — they must either read multiple files or grep blindly. LLM generation of summaries and tags is also expensive; regenerating everything on every run is wasteful when only a few docs changed.

## Key Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Mode detection | CREATE if no `input-index.yaml`; UPDATE otherwise | Matches `extraction-manifest.yaml` pattern |
| Change detection | Git SHA per file, stored in a `files:` block | SHAs separate from `sections:` — no redundant SHA per section |
| SHA for untracked files | `"untracked"` literal | Consistent with `extraction-manifest.yaml`; always re-processes staged-but-uncommitted files |
| Heading depth | H1–H3 only | H4+ is typically formatting noise, not distinct policy topics |
| Heading level encoding | `#`/`##`/`###` prefix in `heading:` value | Eliminates separate `level:` field; self-documenting |
| No-heading fallback | One entry using filename stem as `#` heading | Ensures all files appear in the index |

## YAML Output Structure

```yaml
# Auto-generated by /index-inputs — do not edit manually
# Domain: <domain>
# Generated: YYYY-MM-DD
files:
  input/policy_docs/ADLTC.md:
    sha: "53e9ffe72bee82a7106f78191bcd3c960692bb89"
  "input/policy_docs/apa_manual/441/441-1 EARNED INCOME.md":
    sha: "untracked"

sections:
  - path: input/policy_docs/ADLTC.md
    heading: "# ADLTC"
    summary: "Overview of the Adult Day and Long-Term Care program, its purpose and scope."
    tags: [ADLTC, adult day care, long-term care, APA, eligibility]

  - path: input/policy_docs/apa_manual/441/441-1 EARNED INCOME.md
    heading: "# 441-1 EARNED INCOME"
    summary: "Defines earned income types counted toward APA eligibility, including wages, tips, and self-employment."
    tags: [earned income, wages, tips, self-employment, FICA]

  - path: input/policy_docs/apa_manual/441/441-1 EARNED INCOME.md
    heading: "## Types of Earned Income"
    summary: "Lists specific wage and salary categories that constitute earned income under APA rules."
    tags: [wages, salary, tips, FICA, net earnings]
```

Conventions:
- `files:` block: alphabetically sorted by path; blank line separating it from `sections:`
- `sections:` block: sorted by path alphabetically, then heading appearance order within each file
- Blank line between entries within `sections:` for readability
- All `summary:` and `heading:` values as quoted strings
- `tags:` as inline bracket list `[tag1, tag2, ...]`
- `sha:` values as quoted strings (matching `extraction-manifest.yaml`)

## Technical Considerations

- **LLM-only execution**: No Python tools, no Makefile targets, no new `settings.local.json` permissions needed. Uses Read and Write tools only.
- **Git SHA lookup**: `git log -1 --format="%H" -- domains/<domain>/<domain-relative-path>` per file (repo-relative path); if output is empty (untracked/staged), use `"untracked"`.
- **UPDATE change detection**: Re-process a file if its current SHA differs from the stored SHA in `files:`, or if it's new (not in `files:`). Remove entries for files no longer on disk.
- **Scope**: This plan covers only creating the command. Integration into `/extract-ruleset` (reading the index before scanning docs) is a separate future task.

## File to Create

### `.claude/commands/index-inputs.md`

```markdown
# Index Input Policy Documents

Scan a domain's `input/policy_docs/` directory and produce `specs/input-index.yaml` — a structured index of all policy document sections with LLM-generated summaries and topic tags. The index is the first place to look when searching for which document covers a given policy topic; specific documents can then be read for full detail.

On first run (CREATE), all files are indexed. On subsequent runs (UPDATE), only changed or new files are re-processed using git SHA comparison.

## Input

\```
/index-inputs <domain>
\```

If `<domain>` is not provided, list all `domains/*/input/policy_docs/` directories and prompt the user to choose.

## Pre-flight

Run these checks before doing anything else:

1. **Domain argument provided?**
   - NO → List all directories matching `domains/*/input/policy_docs/` and prompt: "Which domain? (provide as argument)" Then stop.

2. **Domain folder exists?**
   - NO → Print: `"Domain not found: domains/<domain>/"` Then stop.

3. **Input docs present?**
   - `domains/<domain>/input/policy_docs/` missing or contains no `.md` files → Print: `"No input documents found. Add .md files to domains/<domain>/input/policy_docs/ and re-run."` Then stop.

## Mode Detection

\```bash
ls domains/<domain>/specs/input-index.yaml 2>/dev/null
\```

| Result | Mode |
|--------|------|
| File absent | **CREATE** — index all files from scratch |
| File present | **UPDATE** — re-index changed/new files only |

---

## Process — CREATE Mode

### Step 1: Discover all input documents

Glob all `.md` files recursively under `domains/<domain>/input/policy_docs/`. Sort alphabetically by path.

Print: `"Found <N> document(s). Indexing all..."`

### Step 2: Get git SHA for each file

For each file, run:

\```bash
git log -1 --format="%H" -- domains/<domain>/<domain-relative-path>
\```

If the output is empty (file is untracked or staged but not committed), use `"untracked"`.

If any files have `"untracked"` SHA, prompt the user before proceeding:

\```
⚠ <N> file(s) are not yet committed to git:
    input/policy_docs/<file1>.md
    input/policy_docs/<file2>.md
  Untracked files are re-indexed on every UPDATE run (no SHA to compare).
  Commit these files now? [y/n]:
\```

- **y** → Run:
  \```bash
  git add domains/<domain>/input/policy_docs/<file1>.md domains/<domain>/input/policy_docs/<file2>.md
  git commit -m "chore(<domain>): add input policy docs"
  \```
  Then re-fetch the SHA for each committed file (`git log -1 --format="%H" -- ...`) and use the real SHA in the index.
- **n** → Continue. These files get `"untracked"` SHA and will be re-indexed on every future UPDATE run.

### Step 3: Extract sections and generate index entries

For each file (in sorted order):

1. Read the full file content.
2. Extract all H1 (`#`), H2 (`##`), and H3 (`###`) headings in the order they appear. For each heading, collect the text between it and the next heading of equal or higher level — this is the section body.
3. For each section: generate a 1-sentence summary describing the policy content, and 3–5 topic tags (short noun phrases).
4. If a file has **no H1–H3 headings**, emit one entry for the whole file using the filename stem as the heading, prefixed with `#`.

### Step 4: Write `specs/input-index.yaml`

Ensure `domains/<domain>/specs/` exists. Write the full index:

\```yaml
# Auto-generated by /index-inputs — do not edit manually
# Domain: <domain>
# Generated: YYYY-MM-DD
files:
  input/policy_docs/<file1>.md:
    sha: "<sha>"
  input/policy_docs/<file2>.md:
    sha: "<sha>"

sections:
  - path: input/policy_docs/<file1>.md
    heading: "# Heading Text"
    summary: "One sentence describing what this section covers."
    tags: [tag1, tag2, tag3]

  - path: input/policy_docs/<file1>.md
    heading: "## Subheading"
    summary: "One sentence describing what this section covers."
    tags: [tag1, tag2, tag3]
\```

After writing, print:

\```
✓ specs/input-index.yaml written (CREATE).
  <N> file(s) indexed, <M> section(s) total.

To use this index:
  - Consult it first when searching for which document covers a topic
  - Re-run /index-inputs <domain> after adding or modifying input documents
\```

## Process — UPDATE Mode

### Step 1: Read the existing index

Read `domains/<domain>/specs/input-index.yaml`. Extract the current `files:` SHA map and `sections:` list.

### Step 2: Discover current input documents

Glob all `.md` files recursively under `domains/<domain>/input/policy_docs/`. Sort alphabetically by path.

### Step 3: Get current git SHA for each file

For each current file, run:

\```bash
git log -1 --format="%H" -- domains/<domain>/<domain-relative-path>
\```

Empty output → `"untracked"`.

### Step 4: Classify files

Compare current files + SHAs against the stored `files:` block:

| Condition | Action |
|-----------|--------|
| File in index, SHA unchanged | **SKIP** — keep existing sections |
| File in index, SHA changed | **REINDEX** — re-read and regenerate sections |
| File not in index (new) | **REINDEX** — read and generate sections |
| File in index but not on disk (deleted) | **REMOVE** — drop from `files:` and `sections:` |

Print:
\```
Found <N> document(s): <X> changed/new, <Y> unchanged, <Z> deleted.
Re-indexing <X> file(s)...
\```

If any current files have `"untracked"` SHA, prompt the user:

\```
⚠ <N> file(s) are not yet committed to git:
    input/policy_docs/<file1>.md
  Untracked files are re-indexed on every UPDATE run (no SHA to compare).
  Commit these files now? [y/n]:
\```

- **y** → Commit them (same as CREATE mode Step 2 commit flow), then re-fetch real SHAs before proceeding.
- **n** → Continue with `"untracked"` SHA.

### Step 5: Re-index changed and new files

For each file marked REINDEX: follow Steps 2–3 from CREATE mode (read, extract headings, generate summaries and tags). Replace the old sections for that file in the sections list with the newly generated ones.

### Step 6: Write updated `specs/input-index.yaml`

Rebuild the complete index using:
- The updated `files:` block (all current files with current SHAs)
- The merged `sections:` list (preserved unchanged sections + new/updated sections), re-sorted by path then heading order
- Today's date in the `# Generated:` header line

After writing, print:

\```
✓ specs/input-index.yaml updated (UPDATE).
  <X> file(s) re-indexed, <Y> skipped (unchanged), <Z> removed.
  <M> section(s) total.
\```

## Common Mistakes to Avoid

- **Don't use block-style lists for tags** — `tags: [tag1, tag2, tag3]` not `tags:\n  - tag1`
- **Don't use absolute paths** — paths must start with `input/policy_docs/`, not `/Users/...` or `domains/<domain>/`
- **Don't skip the heading prefix** — `heading: "# Title"` not `heading: "Title"`; the `#` characters encode the level
- **Don't merge all sections from a file into one entry** — each H1–H3 heading is its own entry
- **Don't omit the blank line between sections: entries** — entries separated by one blank line for readability
- **Don't omit the blank line between the `files:` and `sections:` blocks**
- **Don't re-index unchanged files in UPDATE mode** — only re-process files whose SHA changed or that are new
```

## Dependencies & Risks

- **No blocking dependencies** — self-contained; produces or updates one file only
- **Risk — heading detection**: Some docs may use setext-style headings (`===`/`---`). Only extract `#`-prefixed markdown headings.
- **Risk — section boundary**: The "section body ends at next heading of equal or higher level" rule can be tricky for H2 sections nested under H1. The summary should cover only the immediate section's content.
- **Risk — untracked files always re-processed**: Staged files always get SHA `"untracked"`, so they're re-indexed every UPDATE run until committed. This is intentional and matches `extraction-manifest.yaml` behavior.
