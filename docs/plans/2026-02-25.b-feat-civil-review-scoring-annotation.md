# feat: Annotate CIVIL rules and computed fields with extraction review scores

## Overview

Add an optional `review:` block to every entry in `rules:` and `computed:` in the CIVIL YAML. The block carries four 1–5 scores (extraction fidelity, source clarity, logic complexity, policy complexity) and an optional `notes:` string. Scores are assigned by Claude inline during Step 4 of `/extract-ruleset` and used at the Step 8 review gate to surface high-risk items first.

## Problem Statement

The `/extract-ruleset` Step 8 human review gate presents rules in sequential order with no signal about which items are high-risk. A reviewer must read every rule with equal attention, even when some are straightforward threshold comparisons and others encode ambiguous legalese with multiple exceptions. This makes review slow and increases the chance that the risky items get the least scrutiny.

## Proposed Solution

During extraction, Claude self-assesses each rule and computed field on four dimensions and writes the result directly into the CIVIL YAML as a `review:` sibling block. At Step 8, the command re-presents items sorted into two buckets:

- **⚠️ HIGH PRIORITY** — any score ≤ 2 or ≥ 4
- **✅ LOW PRIORITY** — all four scores exactly 3

High-priority items include the full source quote, CIVIL expression, score breakdown, a "flagged for" label, and the `notes:` field. Low-priority items are a compact bulleted list.

## Technical Considerations

- `review:` is an unknown key on rule and computed field dicts today. The current validator silently ignores unknown keys — no existing CIVIL files break.
- `tools/transpile_to_opa.py` reads rule dicts by known key names and does not iterate over unknown keys. **Audit required** before merging: confirm the transpiler does not raise on encountering `review:` on a rule or computed field entry. (Expected: no change needed.)
- The schema file (`specs/ruleset_schema.yaml`) is documentation-only (pure `#` comments). Adding the `review:` block is a docs-only change with no parser impact.
- Scores are generated by Claude during extraction — there is no algorithmic scoring. Consistency depends on the rubric being embedded in the command prompt.

## Detailed Implementation

### 1. `specs/ruleset_schema.yaml` — New comment section

Add after the `# RULES` section banner (currently the last section). Follow the existing `# ===...===` banner format:

```yaml
# =============================================================================
# REVIEW BLOCK — Inline extraction scoring (optional, on rules and computed fields)
# =============================================================================
#
# review:                           # optional; present on rules: entries and computed: fields
#   extraction_fidelity: int        # 1–5; how accurately the AI captured the policy intent
#   source_clarity: int             # 1–5; how clear and unambiguous the source policy text was
#   logic_complexity: int           # 1–5; number of conditions, boolean depth, table lookups
#   policy_complexity: int          # 1–5; density of legalese, cross-references, exceptions
#   notes: string                   # optional; explanation for flagged items (any score ≤2 or ≥4)
#
# All four integer fields are required when the block is present.
# notes: is optional but expected for any item where a score is ≤ 2 or ≥ 4.
# Flagging threshold for Step 8 review: any score ≤ 2 OR ≥ 4 → HIGH PRIORITY.
# Scores have no effect on transpilation or OPA evaluation.
#
# Scale reference:
#   1 = very low  (low fidelity / low clarity / trivial / plain English)
#   3 = moderate  (reasonable translation / some ambiguity / a few conditions / some jargon)
#   5 = very high (direct quote match / exact formula / many nested conditions / dense statute)
#
# Example (on a rule):
#
#   rules:
#     - id: "FED-SNAP-DENY-001"
#       kind: deny
#       priority: 1
#       description: "Gross income exceeds 130% FPL"
#       when: "Household.gross_income > gross_limit"
#       then:
#         - add_reason:
#             code: GROSS_INCOME_EXCEEDED
#             message: "Gross income exceeds the 130% FPL limit"
#       review:
#         extraction_fidelity: 5
#         source_clarity: 5
#         logic_complexity: 2
#         policy_complexity: 1
#         notes: "Direct threshold test; formula explicit in 7 CFR § 273.9(a)(1)"
```

### 2. `tools/validate_civil.py` — Add optional review block validation

#### 2a. Helper function (add near the top of validation logic):

```python
def _validate_review_block(review, prefix, errors):
    """Validate an optional review: block on a rule or computed field."""
    if not isinstance(review, dict):
        errors.append(f"{prefix}.review: must be a mapping")
        return
    score_fields = ["extraction_fidelity", "source_clarity", "logic_complexity", "policy_complexity"]
    for field in score_fields:
        val = review.get(field)
        if val is None:
            errors.append(f"{prefix}.review: '{field}' is required")
        elif not isinstance(val, int) or val < 1 or val > 5:
            errors.append(f"{prefix}.review.{field}: must be an integer 1–5, got {val!r}")
    notes = review.get("notes")
    if notes is not None and not isinstance(notes, str):
        errors.append(f"{prefix}.review.notes: must be a string if present")
```

#### 2b. In the existing rules loop (around line 190, inside `for i, rule in enumerate(rules):`):

```python
review = rule.get("review")
if review is not None:
    _validate_review_block(review, f"rules[{i}] ({rule.get('id', '?')})", errors)
```

#### 2c. In the existing computed loop (around line 147, inside `for field_name, field_def in computed.items():`):

```python
review = field_def.get("review")
if review is not None:
    _validate_review_block(review, f"computed.{field_name}", errors)
```

### 3. `.claude/commands/extract-ruleset.md` — Four changes

#### 3a. Add scoring rubric (new section, insert before `## Process — CREATE Mode`)

Add a new `## Scoring Rubric` section. This is the reference Claude uses when writing `review:` blocks.

```markdown
## Scoring Rubric

When writing `review:` blocks, score each rule and computed field on four dimensions using this table. Apply scores independently — a rule can have high fidelity and high complexity simultaneously.

| Score | extraction_fidelity | source_clarity | logic_complexity | policy_complexity |
|-------|---------------------|----------------|------------------|-------------------|
| 1 | Guessed; source is silent on this | Contradictory or absent from source | Single boolean or comparison | Plain everyday English |
| 2 | Inferred with low confidence | Vague; multiple reasonable interpretations | 2–3 conditions, no table lookups | Some jargon or implicit cross-refs |
| 3 | Reasonable translation with minor gaps | Reasonably clear with minor ambiguity | 4–6 conditions or 1 table lookup | Moderate legalese or defined terms |
| 4 | Strong match to source text | Clear but uses statutory defined terms | 7–9 conditions or 2+ table lookups | Dense statutory language or CFR references |
| 5 | Direct quote or explicit formula | Exact thresholds/formulas stated verbatim | 10+ conditions, nested booleans, multiple tables | Exceptions-to-exceptions, multi-CFR cross-refs |

**Special cases:**
- Structural allow rules (`when: "true"`) always score `logic_complexity: 1`. Score `extraction_fidelity` and `source_clarity` based on whether the policy explicitly states the default-allow logic or leaves it implicit.
- `notes:` is required for any item where any score is ≤ 2 or ≥ 4. For all-3 items, `notes:` may be omitted.
```

#### 3b. Update Step 4 CIVIL template

In the `rules:` example in Step 4, append a `review:` block after `then:`:

```yaml
    then:
      - add_reason:
          code: "MACHINE_CODE"
          message: "Human-readable explanation"
          citations:
            - label: "7 CFR § 273.9(a)(1)"
              url: "https://..."
              excerpt: "Brief excerpt"
    review:                          # assign scores while policy text is in context
      extraction_fidelity: <1-5>
      source_clarity: <1-5>
      logic_complexity: <1-5>
      policy_complexity: <1-5>
      notes: "<explain any score ≤2 or ≥4>"  # omit if all scores are 3
```

Similarly in the `computed:` example, append after `expr:` / `conditional:`:

```yaml
  <field_name>:
    type: money
    description: "..."
    expr: "<CIVIL expression>"
    review:
      extraction_fidelity: <1-5>
      source_clarity: <1-5>
      logic_complexity: <1-5>
      policy_complexity: <1-5>
      notes: "<explain any score ≤2 or ≥4>"  # omit if all scores are 3
```

Add this instruction paragraph after the CIVIL template block:

> **Scoring:** Assign `review:` blocks to every entry in `rules:` and `computed:` as you draft them. Use the Scoring Rubric above. Write scores while the source policy text is in context — do not defer to a separate pass.

#### 3c. Replace Step 8 (CREATE mode)

Replace the current Step 8 body (lines 282–293) with:

```markdown
### Step 8: Human Review Gate

Partition all `rules:` entries and `computed:` fields into two groups based on their `review:` scores:

- **HIGH PRIORITY** — any score ≤ 2 or ≥ 4
- **LOW PRIORITY** — all four scores exactly 3

Present HIGH PRIORITY items first, followed by the LOW PRIORITY compact list.

**HIGH PRIORITY format** (one block per item):

```
─────────────────────────────────────────────────────────────────
⚠️  <rule-id or "computed: <field_name>">
    Scores: fidelity:<N> clarity:<N> logic:<N> policy:<N>
    Flagged for: <comma-separated list of triggered dimensions>
    Policy: "<exact source sentence(s)>"
    CIVIL:  <when: expression or expr:/conditional:>
    Notes:  <notes field content, or "(none)" if omitted>
─────────────────────────────────────────────────────────────────
```

For `Flagged for:`, list each triggered dimension with a short label:
- Score ≤ 2 on `extraction_fidelity` → "low extraction fidelity"
- Score ≤ 2 on `source_clarity` → "low source clarity"
- Score ≥ 4 on `logic_complexity` → "high logic complexity"
- Score ≥ 4 on `policy_complexity` → "high policy complexity"

Example:
```
─────────────────────────────────────────────────────────────────
⚠️  FED-SNAP-DENY-003
    Scores: fidelity:2 clarity:2 logic:3 policy:4
    Flagged for: low extraction fidelity, low source clarity, high policy complexity
    Policy: "Households in which a member is an ineligible alien are ineligible,
             unless an exception under 7 CFR § 273.11(c) applies."
    CIVIL:  Applicant.citizenship_status == "ineligible_alien" &&
            !exists(Applicant.exemption_code)
    Notes:  "'unless' clause likely needs a separate allow rule for exemption paths;
             only a single exemption code check is implemented here."
─────────────────────────────────────────────────────────────────
```

**LOW PRIORITY format** (compact list):

```
✅  LOW PRIORITY ITEMS (<N> items — all scores = 3)
    • FED-SNAP-DENY-001: Gross income exceeds 130% FPL limit
    • FED-SNAP-DENY-002: Net income exceeds 100% FPL limit
    • computed: gross_income — total household gross monthly income
    ...
```

**Edge cases:**
- If ALL items are HIGH PRIORITY, omit the LOW PRIORITY section entirely.
- If ALL items are LOW PRIORITY, omit the HIGH PRIORITY section and note: "All items scored 3/3/3/3 — no flagged items."

**Summary header** (always show):
```
Review summary: <H> HIGH PRIORITY, <L> LOW PRIORITY  (<total> items total)
```

Ask: "Does this translation correctly capture the policy intent? Any rules missing or incorrect?"

**On rejection:** Re-extract the specific disputed rule or computed field, re-validate, recompute its `review:` scores, then re-present the full review gate. Do not proceed to transpilation until the user confirms.
```

## Dependencies & Risks

- **No runtime risk** — `review:` blocks are metadata only; they do not affect Rego output or OPA evaluation.
- **Subjectivity risk** — Scores are LLM-generated and will vary across runs and models. The rubric mitigates this but cannot eliminate it. This is acceptable for a triage aid; scores are not authoritative.
- **Threshold noise** — The "any score ≥ 4" trigger means complex-but-correctly-extracted rules are flagged. This is intentional (complexity warrants a review pass) but may surprise reviewers expecting only low-confidence items in the HIGH PRIORITY bucket. The "Flagged for" label on each item makes the reason transparent.
- **Future: `human_reviewed` flag** — Deferred. Currently there is no persistent record that a human verified a HIGH PRIORITY item. This becomes relevant once UPDATE mode is used repeatedly on an evolving policy doc.
